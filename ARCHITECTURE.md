# Вассал-Литигатор — Архитектура плагина

> Плагин для помощи юристу в ведении судебных дел: от приёма документов клиента до кассационной жалобы.

**Версия архитектуры:** 0.5.0 (адаптирована под cc-edition v1.0.0)
**Дата:** 2026-04-20
**Ревизия:** 2026-04-22

> **Примечание про cc-edition:** документ унаследован из `vassal-litigator` v0.5.4 (Cowork edition) и описывает ту же архитектуру, применённую в Claude Code. В `vassal-litigator-cc` удалены Cowork-специфичные обходы: вместо «копия в `На удаление/` + обнуление через `: >`» используется обычный `rm`; резолвер `$CODEX_COMPANION` упрощён с 5-tier до 3-tier (убраны пути `remote-plugins` из sandbox-сессий Cowork). Историю изменений см. в `README.md` §«Отличия от vassal-litigator (Cowork edition)» и `CHANGELOG.md`.

## 1. Общая концепция

### 1.1. Роли

- **Сюзерен** — юрист-пользователь, который ведёт дело и принимает решения.
- **Вассал** — агент (Claude), который выполняет рутинные и аналитические задачи по указаниям Сюзерена.

### 1.2. Принципы

1. **Агент не принимает решений за юриста.** Вассал анализирует, систематизирует, предлагает — но каждое юридически значимое действие утверждает Сюзерен.
2. **Preview → Apply.** Все команды, изменяющие файловую структуру или карточку дела, работают в два шага: сначала показывают план изменений (preview), затем выполняют только после подтверждения Сюзерена (apply). Это касается: `intake`, `add-evidence`, `add-opponent`, `update-index` (при удалении записей), обновлений `case.yaml` из транскрипций.
3. **Неизменяемость исходников.** Оригинальные файлы, полученные от клиента или оппонента, сохраняются нетронутыми в `.vassal/raw/`. Все переименования и перемещения выполняются как копирование в новое место. Оригинал никогда не удаляется и не перезаписывается.
4. **Экономия токенов через md-зеркала.** Каждый документ дела при обработке получает полнотекстовое md-представление (канонизированный текст). Агент по умолчанию работает только с md-зеркалами, к оригиналам обращается в исключительных случаях (верификация цитат, проверка визуальных реквизитов). Md-зеркало — это кэш извлечённого текста, а не замена юридически значимого оригинала.
5. **Единый индекс дела.** Все скиллы и команды читают и пишут в один индекс — `.vassal/index.yaml`. Это «память» агента по делу.
6. **Маршрутизация моделей.** Файловые операции (OCR, зеркала, индексация, переименование, перемещение) выполняет Codex medium с флагом `--write`. Аналитические задачи — Opus. Оркестрация, preview, взаимодействие с Сюзереном — Sonnet (основной поток). Контрольное ревью аналитических выходов — Codex xhigh (read-only). Построение таймлайна — Codex high. Visualize через `$imagegen` (gpt-image-1.5) — sidecar-only.
7. **Идемпотентность.** Повторный запуск команды не дублирует файлы и записи в индексе; агент проверяет, что уже сделано.
8. **Аудируемость правовых выводов.** Каждый юридический вывод агента (сроки, подсудность, позиция) сопровождается структурой: заключение, фактическое основание, правовое основание, источники, степень уверенности, недостающие данные.

### 3.1. Правила именования процессуальных папок

Шаблон: `{ГГГГ-ММ-ДД} {Сторона} - {название процессуального документа или действия}`

Примеры:
- `2026-03-01 Истец - исковое заявление`
- `2026-03-15 Ответчик - отзыв на иск`
- `2026-04-01 Истец - пояснения по делу`
- `2026-04-10 к судебному заседанию` (если неясно, к какому документу)

Процессуальные папки создаются скиллом `add-opponent` в корне дела. Материалы клиента в них не попадают — они раскладываются хронологически в `Материалы от клиента/` (см. §3.2).

### 3.2. Хронологическая раскладка «Материалы от клиента/»

Правило: **раскладка по хронологии документов, не по тематике**.

- **Одиночный документ** (нет приложений) → файл `Материалы от клиента/ГГГГ-ММ-ДД Отправитель Описание.расш`. Папка вокруг не создаётся.
- **Комплект** (документ с приложениями — договор + приложения, претензия + расчёт, акт + сопровод) → папка `Материалы от клиента/ГГГГ-ММ-ДД Отправитель Описание/`. Внутри — головной документ с тем же именем и приложения по шаблону `Приложение NN — Краткое описание.расш`.
- **Сирота без даты и без привязки к хронологии** (переписка, случайные скриншоты, служебное) → тематическая папка `Материалы от клиента/Без даты — <Тема>/`. Флаг `needs_manual_review: true` — Сюзерен дочищает вручную.
- **Скриншот/картинка** (`.png`, `.jpg`, `.tif`, `.bmp`, `.heic`) → конвертируется apply-фазой в PDF через `scripts/image_to_pdf.py`, оригинал остаётся в `.vassal/raw/`.
- **Архив-оригинал** (`.zip`, `.rar`, `.7z`, `.tar*`) → `.vassal/raw/intake-ГГГГ-ММ-ДД/` и не индексируется; в `index.yaml` попадает его распакованное содержимое.

Запрещено: тематические папки (`Договоры/`, `Переписка/`, `Акты/`) для файлов с привязкой к хронологии. Тематические папки допустимы **только** для сирот по шаблону «Без даты — <Тема>».

Полная спецификация — `shared/conventions.md` §«Хронологическая раскладка».

## 4. Формат карточки дела (case.yaml)

```yaml
# .vassal/case.yaml
version: 2

case:
  number: "А33-12345/2026"
  court: "Арбитражный суд Красноярского края"
  judge: "Иванова А.Б."
  category: "экономические споры"  # или: банкротство, административные и т.д.
  subject: "Взыскание задолженности по договору поставки"
  status: active  # active | suspended | completed | archived

  # Даты
  filing_date: 2026-01-15
  next_hearing: 2026-04-15
  next_hearing_source: user       # user | transcript | kad — кто установил дату

  # Стороны (каждая имеет стабильный party_id для ссылок)
  parties:
    - party_id: plaintiff-01
      role: Истец
      name: "ООО «Ромашка»"
      short_name: "Ромашка"
      inn: "2466123456"
      ogrn: "1022466123456"
      address: "г. Красноярск, ул. Ленина, 1"
      representative: null

    - party_id: defendant-01
      role: Ответчик
      name: "ООО «Лютик»"
      short_name: "Лютик"
      inn: "2466654321"
      ogrn: "1022466654321"
      address: "г. Красноярск, ул. Мира, 5"
      representative: null

  # Наш клиент
  our_client:
    party_id: defendant-01        # ссылка на стабильный ID стороны

  # Представитель
  representative:
    name: "Иванов Иван Иванович"
    short_name: "И. И. Иванов"
    position: "Представитель"
    power_of_attorney: "Доверенность от 01.01.2026"
    address: "г. Москва, ул. Примерная, 1, оф. 10"
    phone: "+7 (999) 000-00-00"
    email: "lawyer@example.com"

  # Суммы
  amounts:
    claim: 1500000.00
    court_fee: 28000.00

  # Требования обязательного досудебного порядка
  pretrial:
    required: true
    required_source: agent        # user | agent | document — кто определил
    complied: true
    complied_source: agent        # user | agent — кто подтвердил
    pretrial_date: 2025-12-01

  # Хронология ключевых событий (пополняется)

    - date: 2026-01-15
      event: "Подача искового заявления"
      source: user                # user | transcript | agent
    - date: 2026-02-01
      event: "Принятие иска к производству"
      source: user
    - date: 2026-03-01
      event: "Предварительное заседание"
      source: user
```

## 5. Формат индекса документов (index.yaml)

Индекс — это единый реестр всех документов дела. Каждый документ получает уникальный ID (`doc-NNN`), который используется для связи с md-зеркалом.

```yaml
# .vassal/index.yaml
version: 2
last_updated: 2026-03-26T14:30:00

documents:
  - id: doc-001
    file: "Материалы от клиента/2025-06-01 ООО Ромашка Договор поставки №47/2025-06-01 ООО Ромашка Договор поставки №47.pdf"
    mirror: "mirrors/doc-001.md"
    type: договор
    title: "Договор поставки №47"
    date: 2025-06-01
    parties:
      - "ООО «Ромашка»"
      - "ООО «Лютик»"
    summary: "Договор поставки строительных материалов. Общая сумма 1 500 000 руб. Срок поставки — 30 дней с момента оплаты."
    seal: true
    signature: true
    completeness: full       # full | partial | fragment
    quality: high             # high | medium | low (качество извлечения текста)
    needs_manual_review: false  # true, если OCR низкого качества
    pages: 12
    source: client            # client | opponent | court | public
    added: 2026-03-26
    processed_by: haiku       # модель, которая обработала документ

    # Происхождение файла
    origin:
      name: "договор поставки.pdf"         # исходное имя файла, как получено
      received: 2026-03-26                   # дата получения
      intake_batch: "intake-2026-03-26"      # пакет приёма (папка в raw/)

    # Целостность зеркала
    mirror_stale: false       # true, если оригинал изменился после создания зеркала
    last_verified: 2026-03-26

    # Бандлы (группы связанных документов)
    bundle_id: null            # ID бандла, если документ входит в группу
    parent_id: null            # ID родительского документа (для приложений)
    role_in_bundle: null       # main | appendix | exhibit

    # Процессуальный статус
    filing_status: null        # null | draft | submitted | filed
    filing_folder: null        # путь к процессуальной папке, если подан

    tags:
      - основной-договор
      - поставка

  - id: doc-002
    file: "Материалы от клиента/Переписка/2025-09-15 ООО Ромашка Претензия.pdf"
    mirror: "mirrors/doc-002.md"
    type: претензия
    title: "Претензия о нарушении сроков поставки"
    date: 2025-09-15
    parties:
      - from: "ООО «Ромашка»"
        to: "ООО «Лютик»"
    summary: "Претензия Ромашки о нарушении сроков поставки по Договору №47. Требование: поставить товар в 10-дневный срок."
    seal: true
    signature: true
    completeness: full
    quality: medium
    needs_manual_review: false
    pages: 2
    source: client
    added: 2026-03-26
    processed_by: haiku
    origin:
      name: "претензия скан.pdf"
      received: 2026-03-26
      intake_batch: "intake-2026-03-26"
    mirror_stale: false
    last_verified: 2026-03-26
    bundle_id: null
    parent_id: null
    role_in_bundle: null
    filing_status: null
    filing_folder: null
    tags:
      - досудебный-порядок
      - претензия

# Бандлы — группы связанных документов (документ + приложения)
bundles:
  # Пример:
  # - id: bundle-001
  #   title: "Исковое заявление с приложениями"
  #   main_doc: doc-010
  #   members: [doc-010, doc-011, doc-012]

# Счётчик для генерации следующего ID
next_id: 3
next_bundle_id: 1
```

### 5.1. Типы документов (taxonomy)

Поле `type` — из фиксированного перечня:

**Договорные:** `договор`, `дополнительное-соглашение`, `приложение-к-договору`, `спецификация`, `акт`, `накладная`, `счёт`, `счёт-фактура`

**Судебные:** `определение`, `решение`, `постановление`, `протокол-заседания`

## 6. Формат MD-зеркала

Каждый документ дела получает полнотекстовое md-зеркало (канонизированное текстовое представление) в `.vassal/mirrors/`. Имя файла — `{id}.md` (например, `doc-001.md`). MD-зеркало — это кэш извлечённого текста для экономии токенов, а не замена юридически значимого оригинала.

```markdown
---
id: doc-001
source: "Материалы от клиента/Договоры/2025-06-01 ООО Ромашка Договор поставки №47.pdf"
type: договор
title: "Договор поставки №47"
date: 2025-06-01
pages: 12
extraction_method: pdf-text    # pdf-text | ocr | docx-parse | image-read | manual
extraction_model: haiku
extraction_date: 2026-03-26
confidence: high               # high | medium | low (качество извлечения)
---

# Договор поставки №47

## от 01 июня 2025 года

**ООО «Ромашка»**, именуемое в дальнейшем «Покупатель», в лице генерального директора Петрова И.И., действующего на основании Устава, с одной стороны, и **ООО «Лютик»**, именуемое в дальнейшем «Поставщик», в лице генерального директора Сидорова А.А., действующего на основании Устава, с другой стороны, заключили настоящий Договор о нижеследующем:

## 1. Предмет договора

1.1. Поставщик обязуется поставить, а Покупатель — принять и оплатить строительные материалы (далее — «Товар») в ассортименте, количестве и по ценам, указанным в Спецификациях, являющихся неотъемлемой частью настоящего Договора.

[... полный текст документа ...]

---
<!-- EXTRACTION NOTES (технические пометки о качестве извлечения) -->
<!-- Страница 7: таблица извлечена приблизительно, проверить оригинал -->
<!-- Страница 11: подпись и печать — визуальная проверка по оригиналу -->
```

### 6.1. Правила создания md-зеркал

1. **Полный текст** — весь текст документа переносится в md, с сохранением структуры (заголовки, пункты, таблицы).
2. **Метаданные в frontmatter** — id, источник, метод извлечения, дата, уверенность.
3. **HTML-комментарии** — только технические пометки о качестве извлечения (проблемные страницы, неточности OCR). Юридические выводы и аналитика агента хранятся в `analysis/` и `positions/`, а не в зеркале.
4. **Таблицы** — конвертируются в markdown-таблицы.
5. **Изображения** — не переносятся; вместо них ставится `[Изображение: описание]`.
6. **Нечитаемые фрагменты** — помечаются `[Нечитаемо: описание проблемы]`.
7. **Нечитаемые документы** — если OCR даёт низкое качество, документ всё равно получает зеркало с пометкой `confidence: low` и `needs_manual_review: true` в индексе. Агент должен предупредить Сюзерена о таких документах.

## 7. Маршрутизация: OCR pipeline и модели

### 7.1. OCR и извлечение текста (отдельный pipeline)

OCR концептуально отделён от LLM. Сначала программное извлечение, потом LLM для интерпретации.

**Уровень 1 — Программный OCR (бесплатно, детерминированно):**
- PDF с текстовым слоем → `pdftotext` / `pymupdf`
- DOCX → `python-docx` парсинг
- Изображения / сканы без текста → `tesseract` + `ocrmypdf`
- Реализация: `scripts/extract_text.py` + `scripts/setup.sh` (установка tesseract раз за сессию)

**Уровень 2 — MCP-сервер OCR (целевое решение):**
- Локальный MCP-сервер с tesseract, работающий на машине юриста
- Инструменты: `ocr_file(path) → text`, `extract_pdf_text(path) → text`, `convert_image_to_pdf(path) → path`
- Преимущества: не нужна переустановка каждую сессию, возможность подключить облачный OCR для сложных сканов
- Описывается в `.mcp.json` как опциональный коннектор

**Уровень 3 — LLM fallback:**
- Если программный OCR даёт `confidence: low` — отправить страницу Haiku-субагенту для дочистки
- Для изображений, которые tesseract не берёт (рукописный текст, плохое качество) — Haiku vision

### 7.2. Маршрутизация моделей

Модели распределяются по роли и уровню риска ошибки.

| Задача | Модель | Обоснование |
|--------|--------|-------------|
| OCR, извлечение текста, создание md-зеркал | Codex medium (`--write`) | Файловый исполнитель; Python-скрипты вызываются внутри Codex |
| Переименование файлов, перемещение по папкам | Codex medium (`--write`) | Тот же файловый пайплайн |
| Обновление `index.yaml`, обогащение summary | Codex medium (`--write`) | YAML-мутации + YAML-валидация |
| Построение таймлайна (Хронология дела.md) | Codex high (`--write`) | Требует чтения многих зеркал и case.yaml |
| Визуализация через `$imagegen` (sidecar PNG) | Codex medium (`--background`) → gpt-image-1.5 | Branch A: companion --background; Branch B: codex exec direct |
| Контрольное ревью аналитических выходов | Codex xhigh (read-only) | Аудируемый формат BLOCKING; не пишет файлы |
| Оркестрация, preview, взаимодействие с Сюзереном | Sonnet (основной поток) | Git, UX, мета-контроль |
| Первичный правовой анализ | Opus | Глубокий юридический анализ |
| Выработка позиции | Opus | Стратегическое мышление |
| Подготовка к заседанию (документы) | Opus | Аргументация, red team/blue team |
| Анализ транскрипции (структурирование) | Sonnet | Разбивка, очистка |
| Анализ транскрипции (глубокий) | Opus | Юридический анализ |
| Генерация процессуальных документов | Opus | Юридическое письмо |

### 7.3. Реализация маршрутизации

Маршрутизация реализована через `skills/codex-invocation/SKILL.md` — единый контракт диспатча Codex.

Файловые скиллы (intake, catalog, update-index, add-evidence, add-opponent):
- Claude-main (Sonnet): preview — read-only, 0 файловых операций
- После подтверждения Сюзерена: диспатч Codex medium `--write` с промптом из `prompts/file-executor-*.md`
- Codex сам вызывает Python-скрипты: `python3 [PLUGIN_ROOT]/scripts/extract_text.py`

Аналитические скиллы (legal-review, build-position, prepare-hearing, draft-judgment, appeal, cassation):
- Opus: черновик (Фаза 1-2)
- Codex xhigh: контрольное ревью (Фаза 3.5, read-only)
- При REVIEW_BLOCKING: Сюзерен выбирает из 3 опций

Таймлайн: Codex high `--write` строит Хронология дела.md + обновляет case.yaml.timeline

Визуализации: Codex medium `--background` с `$imagegen` → PNG в `.vassal/visuals/`

### 8.1. intake — Приём и систематизация материалов клиента

**Тип:** скилл + команда (`/vassal-litigator:intake`)
**Модель:** Sonnet (оркестрация: сборка промптов, показ плана Сюзерену, верификация) + Codex medium (всё остальное: OCR, распаковка, классификация, раскладка, индексация)
**Вход:** Файлы в `Входящие документы/`
**Выход:** Хронологическая раскладка в `Материалы от клиента/` (одиночные файлы + комплекты + сироты `Без даты — <Тема>/`), оригиналы в `.vassal/raw/intake-ГГГГ-ММ-ДД/`, обработанные файлы из `Входящие документы/` удаляются через `rm` (оригиналы архивированы в `.vassal/raw/`), обновлённый `.vassal/index.yaml` с записями для каждого файла (включая членов комплектов с `bundle_id`/`role_in_bundle`/`parent_id`).

**Зависимости:** `init-case` (должна существовать `.vassal/case.yaml` и `.vassal/index.yaml`). Codex companion доступен — иначе intake блокируется.

**Контракт:** `plan → review → (revise) → apply → verify`.

**Разделение труда:** Claude-main **не читает** исходники клиента, не запускает OCR, не предлагает имена/doc-ID/bundle_id. Всю черновую работу делает Codex medium в двух стадиях.

**Алгоритм:**

**Фаза 1 — Plan (Codex medium, без `--write`):**
1. Claude-main собирает промпт из `prompts/_preamble.md` + `prompts/file-executor-intake-plan.md`, подставляет переменные (`case_root`, `plugin_root`, `batch_name`, `plan_path`, `work_dir`, `next_id_hint`, пустой `revise_feedback`).
2. Диспатч через `codex-companion.mjs task --background --effort medium`.
3. Codex снимает снимок `Входящие документы/`, распаковывает архивы в `.vassal/work/intake-<timestamp>/archives/`, прогоняет OCR (`extract_text.py`), классифицирует (дата/отправитель/описание/doc_type/parties/summary/флаги комплекта), строит хронологическую раскладку, назначает `doc-ID` от `next_id_hint` и записывает план в `.vassal/plans/intake-<timestamp>.md`.
4. Claude-main мониторит до `completed`, читает отчёт, проверяет `PLAN_PATH`, `FILES_PLANNED`, `BUNDLES_PLANNED`, `ORPHANS_PLANNED`.

**Фаза 2 — Review Сюзереном:**
5. Claude-main читает `{{plan_path}}` целиком и показывает Сюзерену как есть — markdown-таблицу файлов, секции «Комплекты», «Сироты без даты», «Конверсии изображений → PDF», «Не обработанные архивы», «Проверки плана». Короткий сопроводительный комментарий на 2-3 строки с числами.
6. Варианты ответа: подтверждение → Фаза 3; правки → Фаза 2b; отмена → остановиться.

**Фаза 2b — Revise (при правках Сюзерена):**
7. Claude-main пересобирает promt plan-фазы с заполненным `revise_feedback` (буквальный текст правок).
8. Codex перезаписывает `{{plan_path}}`, обновляет `{{work_dir}}/` при необходимости.
9. Возврат к Фазе 2. Повторяется сколько нужно.

**Фаза 3 — Apply (Codex medium `--write`):**
10. Claude-main собирает промпт из `prompts/_preamble.md` + `prompts/file-executor-intake-apply.md`, подставляет переменные (`plan_path`, `work_dir`, `plan_timestamp`, `batch_name`).
11. Диспатч: `task --background --write --effort medium`.
12. Codex буквально исполняет план: копирует оригиналы в `.vassal/raw/intake-ГГГГ-ММ-ДД/`, конвертирует скриншоты в PDF через `scripts/image_to_pdf.py`, создаёт md-зеркала в `.vassal/mirrors/`, раскладывает файлы в `Материалы от клиента/` (одиночные файлы без папок, комплекты в папках, сироты в `Без даты — <Тема>/`), обновляет `.vassal/index.yaml` с `bundle_id`/`role_in_bundle`/`parent_id` для комплектов и новым `next_id`, пишет строку в `history.md`, чистит `Входящие документы/` через `rm` (оригиналы уже в `.vassal/raw/`), архивирует план в `.vassal/codex-logs/<plan_timestamp>-intake-plan.md` и удаляет активный план и work-дир (`rm`/`rm -rf`).

**Фаза 4 — Verify (Claude-main):**
13. Читает отчёт apply: статус `DONE`/`DONE_WITH_CONCERNS`, `FILES_INGESTED` = `FILES_PLANNED` (с поправкой на архивы-оригиналы, которые не индексируются), `INDEX_VALIDATION` валиден, `{{plan_path}}` обнулён, `PLAN_ARCHIVED` указывает реальный путь в `.vassal/codex-logs/`.
14. Сохраняет лог сессии `.vassal/codex-logs/<plan_timestamp>-intake-session.md` (промпт plan + отчёт plan + промпт apply + отчёт apply).
15. Показывает Сюзерену финальное резюме: сколько файлов/комплектов/сирот/конверсий, есть ли `needs_manual_review`.

**Блокер при отсутствии Codex:** intake не выполняется в fallback. Claude-main сообщает Сюзерену, что companion недоступен, и останавливается. Юрист специально выделил эту границу ответственности — черновую работу делает Codex medium, Claude-main её не подменяет.

### 8.3. legal-review — Первичный правовой анализ

**Тип:** скилл + команда (`/vassal-litigator:legal-review`)
**Модель:** Opus (основной анализ)
**Вход:** `case.yaml`, `index.yaml`, вводная от клиента/Сюзерена, md-зеркала
**Выход:**
- Единый документ для Сюзерена: `{case-root}/Правовое заключение.md`
- Аналитические блоки для агента: `.vassal/analysis/{qualification.md, deadlines.md, pretrial.md, jurisdiction.md, completeness.md, questions-to-client.md}`

**Зависимости:** `catalog` (полный индекс с саммари)

**Алгоритм:**
1. Прочитать `case.yaml`, `index.yaml`.
2. Прочитать вводную от клиента (из текста команды или файла).
3. Загрузить md-зеркала ключевых документов (договор, претензия, иск).
4. Через Opus-субагент выполнить анализ. **Каждый вывод** оформляется в аудируемом формате:
   ```
   **Вывод:** [заключение]
   **Фактическое основание:** [на чём основано]
   **Правовое основание:** [нормы, статьи]
   **Источники:** [doc-001, doc-003]
   **Уверенность:** высокая | средняя | низкая
   **Недостающие данные:** [чего не хватает для уверенного вывода]
   ```

   **a) Ответы на вопросы Сюзерена** (→ `.vassal/analysis/answers.md` и в case-root документ)
   - Краткие ответы на поставленные вопросы.
   - Если указано расписать подробно — развёрнутый ответ.

   **b) Пресекательные сроки** (→ `.vassal/analysis/deadlines.md` и в case-root документ)
   - Исковая давность (общая — 3 года, специальные).
   - Течение, приостановление, перерыв исковой давности.
   - Процессуальные сроки (обжалование, предъявление ИД).
   - Для каждого срока: начало, истечение, статус (действует/истёк/под вопросом).

   **c) Досудебный порядок** (→ `.vassal/analysis/pretrial.md` и в case-root документ)
   - Требуется ли обязательный досудебный порядок.
   - Если да — проверка соблюдения (претензия направлена? срок выдержан?).

   **d) Подсудность** (→ `.vassal/analysis/jurisdiction.md` и в case-root документ)
   - Определение компетентного суда.
   - Проверка пророгационных / дерогационных оговорок в договоре.
   - Исключительная подсудность.

   **e) Качество и полнота документов** (→ `.vassal/analysis/completeness.md` и в case-root документ)
   - Состояние документов (читаемость, полнота).
   - Отсутствующие документы, которые следует запросить.
   - Документы, требующие нотариального заверения / апостиля и т.д.

   **f) Вопросы клиенту** (→ `.vassal/analysis/questions-to-client.md` и в case-root документ)
   - Перечень вопросов.
   - Перечень документов, которые следует запросить.

5. Собрать весь анализ в один связный документ и сохранить в case root как `Правовое заключение.md`.
6. Обновить `history.md`.

**Модель:** Sonnet (оркестрация) + Codex medium (plan + apply)
**Вход:** Новые файлы в `Входящие документы/` в уже ведущемся деле
**Выход:** Встроенные в существующую хронологию файлы в `Материалы от клиента/` (могут создать новый комплект, либо присоединиться к уже существующему как новые приложения), оригиналы в `.vassal/raw/evidence-ГГГГ-ММ-ДД/`, обновлённый `index.yaml`.

**Зависимости:** `init-case` и выполненный хотя бы один `intake` (индекс не пустой). Codex companion доступен.

**Контракт:** `plan → review → (revise) → apply → verify`. Claude-main не читает исходники и не решает, какие файлы куда класть — всё это делает Codex в plan-фазе.

**Ключевое отличие от intake:** plan-фаза читает существующий `index.yaml` и помимо построения новой раскладки умеет **присоединять** новые доказательства к уже существующим комплектам (например, новый акт к уже заведённому договору-комплекту), используя существующий `bundle_id`. Секция «Присоединения к существующим комплектам» плана — критическая точка ревью Сюзереном, ошибка здесь ломает ранее собранную раскладку.

**Алгоритм:**

1. **Plan** (Codex medium, без `--write`): `prompts/file-executor-add-evidence-plan.md`. Codex снимает снимок `Входящие документы/` и существующей хронологии, распаковывает архивы, OCR, детектирует уже обработанные файлы по `origin.name` + `origin.archive_src`, классифицирует новые, решает встраивание (новый комплект / присоединение к существующему / одиночный / сирота / скриншот → PDF), назначает `doc-ID` от `next_id_hint`, записывает план в `.vassal/plans/add-evidence-<timestamp>.md` с секциями «Новые комплекты», «Присоединения к существующим», «Сироты», «Уже обработанные (пропуск)».
2. **Review Сюзереном**: весь план показывается как есть.
3. **Revise** (при правках): `revise_feedback` + повторный plan.
4. **Apply** (Codex medium `--write`): `prompts/file-executor-add-evidence-apply.md`. Исполняет план буквально: копии в raw, зеркала, раскладка в `Материалы от клиента/` (одиночные, новые комплекты, приложения к существующим комплектам — в уже существующие физические папки), индекс с `source: client`, чистка `Входящие документы/`, архив плана в `codex-logs/`.
5. **Verify** (Claude-main): отчёт + лог сессии.

Идемпотентность встроенная: план-фаза пропускает файлы, уже обработанные в предыдущих `intake-*`/`evidence-*` batches.

**Блокер при отсутствии Codex:** add-evidence не выполняется, Claude-main останавливается.

### 8.5. add-opponent — Приём документов оппонента

**Тип:** скилл + команда (`/vassal-litigator:add-opponent`)
**Модель:** Sonnet (оркестрация) + Codex medium (plan + apply) + Opus (Фаза 5 — экспресс-анализ, отдельно)
**Вход:** Документы от процессуального оппонента в `Входящие документы/`
**Выход:** Процессуальная папка `{ГГГГ-ММ-ДД} {Сторона оппонента} - {тип документа}/` с головным документом + приложениями, оригиналы в `.vassal/raw/opponent-ГГГГ-ММ-ДД/`, индекс обновлён (`source: opponent`, `bundle_id` с `role_in_bundle: head|attachment`).

**Зависимости:** `init-case`. Codex companion доступен для Фаз 1–4. Фаза 5 (Opus) от Codex не зависит.

**Контракт:** `plan → review → (revise) → apply → verify → (опционально Opus-анализ)`.

**Алгоритм:**

1. **Plan** (Codex medium, без `--write`): `prompts/file-executor-add-opponent-plan.md`. Claude-main передаёт `opponent_hint` (список сторон-оппонентов из `case.yaml`). Codex определяет сторону-отправителя, тип головного документа, назначает имя процессуальной папки по шаблону `ГГГГ-ММ-ДД Сторона - Тип документа/`, идентифицирует head + приложения, назначает `bundle_id` и `doc-ID`, записывает план.
2. **Review Сюзереном**: весь план как есть.
3. **Revise** (при правках): `revise_feedback` + повторный plan.
4. **Apply** (Codex medium `--write`): `prompts/file-executor-add-opponent-apply.md`. Исполняет план: raw, зеркала, создание процессуальной папки в корне дела, размещение head и приложений, индекс с `source: opponent`, чистка `Входящие документы/`, архив плана.
5. **Verify** (Claude-main): отчёт + лог сессии.
6. **Фаза 5 — Экспресс-анализ (Opus-субагент, не Codex):** если головной документ процессуальный (отзыв, возражения, пояснения, ходатайство), Claude-main предлагает Сюзерену короткий анализ аргументов через Opus-субагент (`analytical-reviewer` или аналог). Результат сохраняется в `.vassal/analysis/opponent-<plan_timestamp>.md`. Файловая работа сюда не входит.

**Блокер при отсутствии Codex:** Фазы 1–4 не выполняются. Фаза 5 при наличии уже обработанной поставки в деле — доступна (это чистая аналитика Opus).

### 8.6. update-index — Обновление и верификация индекса

**Тип:** скилл + команда (`/vassal-litigator:update-index`)
**Модель:** Sonnet
**Вход:** Текущее состояние папки дела
**Выход:** Обновлённый и верифицированный `index.yaml`

4. Обновить `index.yaml` и сообщить Сюзерену о результатах.

### 8.7. build-position — Выработка правовой позиции

**Тип:** скилл + команда (`/vassal-litigator:build-position`)
**Модель:** Opus
**Вход:** `case.yaml`, `index.yaml`, md-зеркала, вводная Сюзерена
**Выход:**
- Документ для Сюзерена: `{case-root}/{ГГГГ-ММ-ДД} {тип} позиция.md`
- Аналитические блоки для агента (при необходимости переиспользования): `.vassal/positions/{блоки}`

**Зависимости:** `legal-review`

**Алгоритм (Opus-субагент):**
1. Загрузить case.yaml, index.yaml, все файлы из analysis/.
2. Загрузить md-зеркала ключевых документов.
3. Проанализировать (каждый вывод в аудируемом формате — см. legal-review):
   - **Основания иска/защиты** — фактические и правовые.
   - **Сильные стороны позиции клиента** — со ссылками на документы и нормы.
   - **Слабые стороны и уязвимости** — что может использовать оппонент.
   - **Оценка рисков** — вероятность удовлетворения/отказа, по каждому требованию.
   - **Способы усиления позиции** — какие доказательства получить, какие аргументы развить.
4. Сохранить полную позицию в case root: `{ГГГГ-ММ-ДД} {тип} позиция.md` (видна Сюзерену).

### 8.8. prepare-hearing — Подготовка к заседанию

**Тип:** скилл + команда (`/vassal-litigator:prepare-hearing`)
**Модель:** Opus
**Вход:** Всё состояние дела + указания Сюзерена
**Выход:** Правовая позиция к заседанию + проект процессуального документа (.docx)

**Зависимости:** `build-position`, `arbitrum-docx` (для оформления)

**Алгоритм (Opus-субагент):**
1. Загрузить полный контекст дела:
   - case.yaml, index.yaml
   - Все позиции из case root (файлы вида `{ГГГГ-ММ-ДД} {тип} позиция.md`)
   - Все отчёты из `.vassal/hearings/` (если были предыдущие заседания)
   - md-зеркала процессуальных документов обеих сторон
2. Учесть:
   - Ранее подготовленные процессуальные документы Сюзерена.
   - Позицию оппонента из его документов и выступлений.
   - Ремарки и позицию судьи (из отчётов о заседаниях).
3. Работа по принципу **red team / blue team**:
   - **Blue team**: аргументы в пользу нашего клиента.
   - **Red team**: контраргументы, которые выдвинет оппонент, и слабости нашей позиции.
   - **Ответ на red team**: как парировать контраргументы.
4. Сформировать текст процессуального документа.
5. **Оформление через arbitrum-docx** — передать текст скиллу `arbitrum-docx` для генерации .docx по шаблону.
6. Поместить .docx в соответствующую процессуальную папку.
7. Обновить `index.yaml` и `history.md`.

### 8.9. analyze-hearing — Анализ транскрипции заседания

**Тип:** скилл + команда (`/vassal-litigator:analyze-hearing`)
**Модель:** Sonnet (структурирование) + Opus (глубокий анализ)
**Вход:** Транскрипция заседания (текст или файл)
**Выход:** Отчёты в `.vassal/hearings/` + (опционально) `.docx` для клиента

**Зависимости:** `init-case`

**Алгоритм:**
1. Прочитать транскрипцию. Учитывать, что транскрипции могут содержать:
   - Артефакты распознавания речи.
   - Галлюцинации (особенно имена, даты, номера).
   - Пропуски и неразборчивые фрагменты.

2. Через Sonnet-субагент (model: "sonnet") — структурирование:
   - Дата заседания, состав участников.
   - Разбивка на блоки по выступающим.
   - Очистка от артефактов (пометить сомнительные места `[?]`).

3. Сформировать три вида отчётов:

   **a) Внутренний отчёт** (`hearings/{ГГГГ-ММ-ДД}-internal.md`) — через Opus-субагент (model: "opus"):
   - Подробная хронология заседания.
   - Позиция каждой стороны (с цитатами).
   - Вопросы и замечания судьи.
   - Процессуальные решения (отложение, назначение экспертизы и т.д.).
   - Выявленные слабости и сильные стороны (наши и оппонента).
   - Рекомендации к следующему заседанию.

   **b) Отчёт для клиента — краткий** (`hearings/{ГГГГ-ММ-ДД}-client-brief.md`)
   - Дата, суд, состав.
   - Что произошло (2-3 абзаца).
   - Позиция суда (если обозначена).
   - Что решено, следующие шаги.

   **c) Отчёт для клиента — красивый** (`.docx` через arbitrum-docx)
   - Фирменно оформленный вариант отчёта (b).
   - Тип документа: «заключение / аналитический документ» (без шапки).

4. **Preview обновлений case.yaml** — показать Сюзерену предлагаемые изменения:
   - Новые события в timeline (с пометкой `source: transcript`).
   - Дата следующего заседания (если озвучена).
   - Приобщённые документы.
   Транскрипции ненадёжны — обновление карточки дела только после подтверждения.
5. После подтверждения — обновить `case.yaml`, `index.yaml`.
6. Обновить `history.md`.

## 10. Интеграция с arbitrum-docx

**Зависимость, не встраивание.** Плагин `arbitrum-docx` — внешний скилл, который vassal-litigator использует для оформления процессуальных документов в .docx. Логика генерации документов не дублируется внутри vassal-litigator.

**Как это работает:** Claude — единый рантайм. Когда скилл vassal-litigator (например, `appeal` или `draft-judgment`) формирует содержание документа, он указывает в инструкциях: «оформи результат через скилл arbitrum-docx». Claude читает оба SKILL.md и комбинирует: vassal-litigator даёт контекст дела (стороны, суд, содержание), arbitrum-docx — шаблон, стили, форматирование.

**Скиллы, использующие arbitrum-docx:**
- `draft-judgment` — проект судебного акта
- `appeal` — апелляционная жалоба
- `cassation` — кассационная жалоба
- `prepare-hearing` — ходатайства, письменные пояснения
- `analyze-hearing` — отчёт по заседанию (опционально)

## 15. Интеграция с Codex CLI

### 15.1. Общая архитектура

Codex CLI (OpenAI GPT-5 family, OAuth) используется в четырёх ролях:
- **Файловый исполнитель** (Codex medium, двухфазный контракт `plan → apply`): OCR, распаковка архивов, классификация, хронологическая раскладка, зеркала, индексация, перемещение файлов. Plan-фаза — без `--write`, только whitelist-записи в `.vassal/plans/` и `.vassal/work/`. Apply-фаза — с `--write`, буквально исполняет одобренный план.
- **Таймлайн-билдер** (Codex high, `--write`): хронология дела.
- **Визуализатор** (Codex medium, `--background` → `$imagegen`): sidecar PNG.
- **Контрольный ревьюер** (Codex xhigh, read-only): аудируемое ревью аналитики.

Claude-main (Sonnet) остаётся оркестратором: сборка промптов, показ плана Сюзерену, сбор правок Сюзерена для revise, верификация отчётов, git, мета-контроль. Черновую работу по материалам клиента (чтение исходников, OCR, принятие решений о раскладке) Claude-main **не выполняет** — эта граница ответственности зафиксирована юристом в v0.5.4.

### 15.2. Контракт вызова

См. `skills/codex-invocation/SKILL.md`. Стандартный вызов:

```bash
node "$CODEX_COMPANION" \
  task --background [--write] --effort medium|high|xhigh [--fresh|--resume-last] \
  "$PROMPT"
```

`$CODEX_COMPANION` резолвится один раз в сессию по 3-tier fallback (env → канонический `$HOME/.claude/plugins/cache/openai-codex/codex/<ver>/scripts/codex-companion.mjs` → парсинг `installed_plugins.json` по ключу `codex@openai-codex`). См. `skills/codex-invocation/SKILL.md` §«Resolve `$CODEX_COMPANION`».

Monitor-polling: `until/case` фильтр по статусу, `sleep 25` между проверками.

### 15.3. Контракт путей [PLUGIN_ROOT]/[CASE_ROOT]

Claude-main подставляет пути в промпты до диспатча. 3-tier fallback для [PLUGIN_ROOT]:
1. Переменная окружения `VASSAL_PLUGIN_ROOT`
2. `~/.claude/installed_plugins.json` → путь к vassal-litigator
3. Спросить Сюзерена

Обязательная проверка перед диспатчем: `grep -c "{{" prompt.txt → 0`.

### 15.4. Четыре роли Codex

| Роль | Флаги | Промпт | Пример скиллов |
|------|-------|--------|----------------|
| Файловый исполнитель — plan | `--effort medium` (без `--write`, whitelist в промпте) | `prompts/file-executor-*-plan.md` | intake, add-evidence, add-opponent |
| Файловый исполнитель — apply | `--write --effort medium` | `prompts/file-executor-*-apply.md` | intake, add-evidence, add-opponent |
| Файловый исполнитель — одностадийный | `--write --effort medium` | `prompts/file-executor-catalog.md`, `prompts/file-executor-update-index.md` | catalog, update-index (без plan-фазы, работают по детерминированному алгоритму) |
| Таймлайн-билдер | `--write --effort high` | `prompts/timeline-builder.md` | timeline |
| Визуализатор | `--background --effort medium` | `prompts/imagegen-visualizer.md` | visualize |
| Контрольный ревьюер | `--effort xhigh` (без `--write`) | `prompts/analytical-reviewer.md` | legal-review, build-position и др. |

### 15.5. Промпты

Все промпты в `prompts/`:
- `_preamble.md` — базовый контекст: два режима plan/apply для файлового пайплайна, неизменяемость `.vassal/raw/`, whitelist write для plan-фазы (`.vassal/plans/`, `.vassal/work/`, `.vassal/history.md`), финальная очистка apply (архив плана в `codex-logs/`, обнуление work-дира), anti-prompt-injection, формат отчёта.
- Пары plan/apply файлового пайплайна:
  - `file-executor-intake-plan.md` + `file-executor-intake-apply.md`
  - `file-executor-add-evidence-plan.md` + `file-executor-add-evidence-apply.md`
  - `file-executor-add-opponent-plan.md` + `file-executor-add-opponent-apply.md`
- Одностадийные файловые роли (без plan-фазы):
  - `file-executor-catalog.md`, `file-executor-update-index.md`
- `timeline-builder.md` — таймлайн.
- `imagegen-visualizer.md` — визуализации через `$imagegen`.
- `analytical-reviewer.md` — контрольное ревью в аудируемом формате.

Каждый промпт начинается с `{{include _preamble.md}}`, который Claude-main разворачивает перед диспатчем. Версия контракта фиксируется в preamble и синхронизирована с CHANGELOG.

### 15.6. Формат аудируемого ревью (BLOCKING)

```
## BLOCKING N: <тема>
- Категория: срок | норма | логика | формат | фактическое противоречие | фабрикация | процедура
- Замечание: <1-2 предложения>
- Фактическое основание: <буквальная цитата + раздел/строка>
- Правовое основание: <норма или "Нет применимой нормы — логическая/процедурная ошибка">
- Источники (doc-NNN): <IDs из index.yaml или "нет — замечание по внутренней логике">
- Уверенность: высокая | средняя | низкая
- Недостающие данные: <что именно>
- Что проверить/уточнить: <конкретное действие>
```

### 15.7. Визуализации — sidecar-only

PNG файлы (`$imagegen` через Codex) хранятся исключительно в `.vassal/visuals/`. Они:
- Не индексируются в `index.yaml`
- Не встраиваются в процессуальные и аналитические документы автоматически
- Не являются доказательствами и не имеют процессуальной силы
- AI-generated иллюстрации для внутреннего использования юриста

Юрист может добавить PNG в документ вручную — это его сознательное решение.

### 15.8. Fallback при отсутствии Codex

При недоступности openai-codex:
- **Файловые скиллы (`intake`, `add-evidence`, `add-opponent`): блокируются.** Claude-main сообщает Сюзерену, что companion недоступен, и останавливается. Самостоятельное исполнение pipeline запрещено — v0.5.4 явно фиксирует, что черновую работу делает Codex medium, Claude-main её не подменяет. Так было не всегда: в v0.5.0–v0.5.2 Claude-main при отсутствии Codex исполнял pipeline сам (как в v0.4.0); с v0.5.4 это поведение отменено по решению юриста.
- `catalog` и `update-index` (одностадийные файловые роли): можно выполнить напрямую Claude-main-ом, т.к. алгоритм детерминированный и не требует OCR исходников. Лог — в `codex-logs/*-fallback.md`.
- Таймлайн: Claude-main строит хронологию без Codex.
- Визуализации: сообщить Сюзерену, что `$imagegen` недоступен.
- Аналитика: без контрольного ревью xhigh.

### 15.9. Конфиденциальность

Материалы дела (тексты документов, зеркала, метаданные) отправляются в OpenAI через Codex CLI. Использование плагина означает явное согласие юриста на такую передачу. Юрист несёт ответственность за соответствие соглашениям с клиентами.

### 15.10. Версионирование companion

Зависимость: `openai-codex ≥ 1.0.3`. Путь к companion резолвится в `$CODEX_COMPANION` по 3-tier fallback (`skills/codex-invocation/SKILL.md` §«Resolve `$CODEX_COMPANION`»). Канонический путь — `$HOME/.claude/plugins/cache/openai-codex/codex/<ver>/scripts/codex-companion.mjs`. Tier 3 (парсинг `installed_plugins.json` по ключу `codex@openai-codex`) покрывает смену `<ver>` и переопределённый `$HOME`.
