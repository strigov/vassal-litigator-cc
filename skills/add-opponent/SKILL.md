# add-opponent — Приём документов оппонента

Скилл работает по контракту **plan → review → (revise) → apply → verify → (анализ Opus)**. Файловую работу — OCR, распаковку, классификацию, раскладку в процессуальную папку — делает Codex medium. Claude-main промпты собирает и план показывает. Экспресс-анализ аргументов оппонента (Фаза 5) делается отдельно Opus-субагентом после apply — это не файловая работа.

## Предусловия

- Дело инициализировано: `.vassal/case.yaml` существует, в нём определены стороны (чтобы понимать, кто оппонент).
- Папка `Входящие документы/` существует и в ней хотя бы один файл.
- Зависимости установлены (`setup.sh`).
- Codex companion доступен.

## Переменные сессии

- `plan_timestamp` = `ГГГГ-ММ-ДД-ЧЧмм`
- `batch_name` = `opponent-ГГГГ-ММ-ДД`
- `plan_path` = `.vassal/plans/add-opponent-<plan_timestamp>.md`
- `work_dir` = `.vassal/work/add-opponent-<plan_timestamp>/`
- `next_id_hint` = `next_id` из `.vassal/index.yaml`
- `opponent_hint` = краткий список сторон-оппонентов из `case.yaml` (чтобы Codex не тратил токены на угадывание). Формируй строкой вида: `Ответчики: ООО Ромашка, ИП Иванов. Третьи лица: ООО Ландыш.` Если стороны в `case.yaml` не заполнены — пусто, Codex определит по содержимому документов.

Создай пустые `.vassal/plans/` и `.vassal/work/`, если их нет.

## Фаза 1 — Plan (Codex medium)

1. Прочитай `skills/codex-invocation/SKILL.md`.
2. Собери промпт: `prompts/_preamble.md` + `prompts/file-executor-add-opponent-plan.md`. Переменные:
   - `role_id`: `file-executor-add-opponent-plan`
   - `task_name`: `add-opponent plan <batch_name>`
   - `case_root`, `plugin_root`
   - `batch_name`, `plan_path`, `work_dir`, `next_id_hint`, `opponent_hint`
   - `revise_feedback`: пусто при первом запуске
   - `report_contract`: из `_preamble.md`
3. `grep -c "{{" <prompt>` → `0`.
4. Диспатч без `--write`: `task --background --effort medium "..."`.
5. Мониторь по шаблону `until/case loop` с `sleep 25`.
6. Получи отчёт. Проверь: статус `DONE`/`DONE_WITH_CONCERNS`, файл `{{plan_path}}` непустой, поля `OPPONENT_PARTY`, `DOC_TYPE_HEAD`, `FILES_PLANNED`, `ATTACHMENTS_PLANNED` присутствуют.
7. `BLOCKED` / `NEEDS_CONTEXT` — покажи Сюзерену `CONCERNS`, спроси действия. Типичный блокер add-opponent — «поставка без головного документа»: тогда предложи Сюзерену либо вручную обозначить голову, либо вернуть поставку.

## Фаза 2 — Review Сюзереном

1. Прочитай `{{plan_path}}` целиком.
2. Покажи Сюзерену **весь план** как есть (markdown). Сопроводительный комментарий в 2-3 строки: кто оппонент, какой тип документа, куда попадает, сколько приложений.
3. Спроси: «Подтверждаешь? Или правки?»
4. Варианты: подтверждение → Фаза 3; правки → Фаза 2b; отмена → остановиться.

### Фаза 2b — Revise

1. Собери промпт plan с заполненным `revise_feedback`.
2. Те же `plan_path` и `work_dir`.
3. После revise-диспатча вернись к Фазе 2.

## Фаза 3 — Apply (Codex medium `--write`)

1. Собери промпт: `prompts/_preamble.md` + `prompts/file-executor-add-opponent-apply.md`. Переменные:
   - `role_id`: `file-executor-add-opponent-apply`
   - `task_name`: `add-opponent apply <batch_name>`
   - `case_root`, `plugin_root`
   - `batch_name`, `plan_path`, `work_dir`, `plan_timestamp`
   - `report_contract`
2. `grep -c "{{" <prompt>` → `0`.
3. Диспатч: `task --background --write --effort medium "..."`.
4. Мониторь до завершения.
5. Получи отчёт.

## Фаза 4 — Verify

1. Прочитай отчёт. Проверь:
   - статус `DONE` / `DONE_WITH_CONCERNS`
   - `PROC_FOLDER` — путь реально создан
   - `HEAD_DOC_ID` и `ATTACHMENTS` согласованы с планом
   - `INDEX_VALIDATION` — YAML валиден
   - `PLAN_ARCHIVED` — путь в `.vassal/codex-logs/` существует
   - `{{plan_path}}` обнулён
2. Прочитай `.vassal/index.yaml`: новые записи имеют `source: opponent`, корректные `bundle_id`/`role_in_bundle`/`parent_id`.
3. Сохрани лог сессии `.vassal/codex-logs/<plan_timestamp>-add-opponent-session.md`: промпт plan + отчёт plan + промпт apply + отчёт apply.
4. Финальное резюме Сюзерену: `Поставка от <оппонент>: <тип документа>, N приложений, процессуальная папка <...>, needs_manual_review: X`.

## Фаза 5 — Экспресс-анализ (Opus, не Codex)

Эта фаза остаётся у Opus — это аналитика, а не файловая работа.

1. Если головной документ оппонента — процессуальный (отзыв, возражения, пояснения, ходатайство, встречный иск), предложи Сюзерену: `Сделать экспресс-анализ аргументов оппонента?`
2. Если Сюзерен согласен — через Opus-субагент (`analytical-reviewer` или аналогичный):
   - извлеки ключевые доводы оппонента из md-зеркала головного документа
   - сопоставь с нашей позицией, если в деле уже есть позиционные документы (своя жалоба/претензия/исковое)
   - кратко ответь: что нового, что опасно, что легко парируется
   - предложи ответные действия по делу
   - сохрани результат в `.vassal/analysis/opponent-<plan_timestamp>.md`
   - покажи Сюзерену краткую сводку в чате

## Идемпотентность

Plan-фаза сама проверяет по `origin.name` + `origin.archive_src` в предыдущих `opponent-*` batches. Уже обработанные файлы попадают в секцию «Уже обработанные (пропуск)» плана и в индекс не повторно не добавляются, но их оригиналы всё равно копируются в raw и обнуляются из `Входящие документы/`.

## Граничные случаи

- **Документы от нескольких оппонентов в одном пакете** — plan-фаза сама поделит на отдельные процессуальные папки и `bundle_id` по оппонентам; Claude-main смотрит на секцию сводки.
- **Документ от суда** (определение, протокол) попал по ошибке — Codex пометит как `needs_manual_review: true` с заметкой о несоответствии; покажи Сюзерену и предложи переквалифицировать или забрать в отдельный intake.
- **Оппонент прислал копии наших же документов** — детектится по содержимому OCR; плановщик пометит такие как `already_processed: true` либо оставит с заметкой «копия нашего документа» для ручного ревью.

## Блокер при отсутствии Codex

Если Codex companion недоступен (`CODEX_COMPANION_NOT_FOUND`, `ECONNREFUSED`):

- Сообщи Сюзерену: `Codex companion недоступен, add-opponent невозможен. Установи/перезапусти плагин openai-codex и повтори.`
- Не переходи к самостоятельному исполнению.
- Остановись.

Экспресс-анализ Opus (Фаза 5) запуск Codex не требует и может быть выполнен позже, когда apply будет завершён.
