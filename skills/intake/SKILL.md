# Intake — Приём материалов клиента

Скилл работает по контракту **plan → review → (revise) → apply → verify**. Всю черновую работу (чтение исходников, OCR, распаковка архивов, классификацию, построение раскладки) делает Codex medium. Claude-main только собирает промпты, показывает Сюзерену план и верифицирует отчёт — **никогда** не читает исходники клиента, не запускает OCR и не предлагает имена/doc-ID/bundle_id.

## Предусловия

- Дело инициализировано: существует `.vassal/case.yaml`. Если нет — предложи запустить `/vassal-litigator:init-case`.
- Существует папка `Входящие документы/` и в ней хотя бы один файл. Если папки нет или она пуста — сообщи Сюзерену и останови intake (нечего обрабатывать).
- В корне дела существует `.vassal/index.yaml` (после init-case гарантированно есть).
- Установлены зависимости (`tesseract`, `ocrmypdf`, `unzip`, `7z`, `tar`, `unrar`, Python-пакеты). Если `[PLUGIN_ROOT]/scripts/setup.sh` на этой машине ещё не запускался — запусти один раз.
- Доступен Codex companion. Если нет — см. раздел «Блокер при отсутствии Codex» ниже.

## Переменные сессии

Вычисли один раз в начале intake и используй во всех фазах:

- `plan_timestamp` = текущее `ГГГГ-ММ-ДД-ЧЧмм` (локальное время машины)
- `batch_name` = `intake-ГГГГ-ММ-ДД` (только дата, без времени)
- `plan_path` = `.vassal/plans/intake-<plan_timestamp>.md`
- `work_dir` = `.vassal/work/intake-<plan_timestamp>/`
- `next_id_hint` = прочитай `next_id` из `.vassal/index.yaml` (при отсутствии — `1`)

Создай пустые `.vassal/plans/` и `.vassal/work/`, если их нет. Не создавай сам файл плана — это работа Codex plan-фазы.

## Фаза 1 — Plan (Codex medium, whitelist write)

Claude-main собирает промпт и диспатчит plan-фазу. **Не читает содержимое файлов из `Входящие документы/`.**

1. Прочитай `skills/codex-invocation/SKILL.md` (раздел «Как Claude-main собирает prompt»).
2. Собери промпт: `prompts/_preamble.md` + `prompts/file-executor-intake-plan.md`. Подставь переменные:
   - `role_id`: `file-executor-intake-plan`
   - `task_name`: `intake plan <batch_name>`
   - `case_root`: абсолютный путь к делу
   - `plugin_root`: абсолютный путь к плагину (3-tier fallback из `skills/codex-invocation/SKILL.md`)
   - `batch_name`, `plan_path`, `work_dir`, `next_id_hint`
   - `revise_feedback`: пусто при первом запуске
   - `report_contract`: блок `Формат отчёта` из `prompts/_preamble.md`
3. Проверь, что плейсхолдеры раскрыты: `grep -c "{{" <prompt>` → `0`.
4. Диспатч через `codex-companion.mjs`:
   `task --background --effort medium "..."`
   **Без `--write`**: план-фаза пишет только в whitelist (плагин-promptы сами это контролируют). Флаг `--write` в companion подразумевает «любые записи разрешены»; для plan-фазы он не нужен.
5. Мониторь до `status: completed` по шаблону `until/case loop` из `skills/codex-invocation/SKILL.md` с `sleep 25`.
6. Получи отчёт. Убедись:
   - статус на первой строке — `DONE` или `DONE_WITH_CONCERNS`
   - в отчёте присутствуют поля `PLAN_PATH`, `FILES_PLANNED`, `BUNDLES_PLANNED`, `ORPHANS_PLANNED`
   - файл `{{plan_path}}` существует и не пустой
7. Если статус `BLOCKED` / `NEEDS_CONTEXT` — покажи Сюзерену секцию `CONCERNS / BLOCKERS` и спроси, что делать (повторить, дать доп. контекст, отменить intake).

## Фаза 2 — Review Сюзереном

1. Прочитай `{{plan_path}}` целиком.
2. Покажи Сюзерену **весь план**, не пересказывая и не резюмируя в свою сторону. Markdown-таблицы, секции «Комплекты», «Сироты без даты», «Конверсии изображений», «Не обработанные архивы», «Проверки плана» — всё это выводится как есть. Допустимо добавить сопроводительный комментарий 2-3 строки с числами (всего файлов / комплектов / сирот).
3. Спроси: «Подтверждаешь? Если есть правки — напиши что поменять (буквально: разделить комплект, переименовать, переместить конкретный файл и т.п.).»
4. Варианты ответа Сюзерена:
   - **Подтверждение** («ок», «apply», «go», «да»): переходи к Фазе 3.
   - **Правки**: собери текст правок как `revise_feedback` и перейди к Фазе 2b (revise).
   - **Отмена**: останови intake. Не трогай план-файл (он самоуничтожится при следующем plan-запуске; при необходимости можно вручную удалить).

### Фаза 2b — Revise (Codex medium, то же самое + feedback)

1. Собери новый промпт из `prompts/file-executor-intake-plan.md`, но с заполненным `revise_feedback` (буквальный текст правок от Сюзерена).
2. `plan_path` и `work_dir` **те же самые**: Codex перезапишет план и при необходимости обновит рабочую область.
3. Диспатч и мониторинг — как в Фазе 1.
4. Вернись к Фазе 2 (показ обновлённого плана). Количество итераций не ограничено, но практически после 2-3 revise-раундов стоит предложить Сюзерену закрыть intake вручную.

## Фаза 3 — Apply (Codex medium `--write`)

Выполняется только после явного подтверждения Сюзерена в Фазе 2.

1. Собери промпт: `prompts/_preamble.md` + `prompts/file-executor-intake-apply.md`. Подставь переменные:
   - `role_id`: `file-executor-intake-apply`
   - `task_name`: `intake apply <batch_name>`
   - `case_root`, `plugin_root`
   - `batch_name`, `plan_path`, `work_dir`, `plan_timestamp`
   - `report_contract`
2. Проверь плейсхолдеры: `grep -c "{{" <prompt>` → `0`.
3. Диспатч:
   `task --background --write --effort medium "..."`
4. Мониторь по шаблону до завершения.
5. Получи итоговый отчёт.

## Фаза 4 — Verify (Claude-main)

1. Прочитай отчёт Codex. Проверь:
   - статус `DONE` или `DONE_WITH_CONCERNS`
   - `FILES_INGESTED` совпадает с `FILES_PLANNED` из plan-отчёта (с точностью до архивов-оригиналов, которые не индексируются)
   - `INDEX_VALIDATION` показывает валидный YAML
   - `PLAN_ARCHIVED` указывает реальный путь в `.vassal/codex-logs/`
   - `{{plan_path}}` обнулён (`wc -c` = 0)
2. Прочитай `.vassal/index.yaml`, убедись, что новые записи имеют все обязательные поля (см. `shared/index-schema.yaml`).
3. Если в `Входящие документы/` остались файлы, которых не было в плане, — Codex сохранил их корректно; если остались файлы **из** плана (Codex не обнулил) — это аномалия, покажи Сюзерену.
4. Сохрани лог сессии: `.vassal/codex-logs/<plan_timestamp>-intake-session.md` — промпт plan + отчёт plan + промпт apply + отчёт apply. (Копия самого плана уже сохранена Codex-ом как `<plan_timestamp>-intake-plan.md` на шаге 9 apply.)
5. Покажи Сюзерену финальное резюме: `N файлов обработано, M комплектов, S сирот, K изображений → PDF. needs_manual_review: X`.

## Имена, папки, идемпотентность

- Формат нового имени файла, правила хронологической раскладки, политика тематических папок (только для сирот) — единый источник: `shared/conventions.md` §«Хронологическая раскладка».
- `doc-ID` всегда назначает plan-фаза, исходя из `next_id` из `index.yaml`. Claude-main IDs не придумывает.
- Идемпотентность: если plan-фаза обнаружит, что исходный файл уже присутствует в `index.yaml` по `origin.name` + `origin.archive_src` + `batch`, она пометит его как `already_processed` в плане и пропустит в apply. Таблица плана всё равно покажет такой файл, чтобы Сюзерен видел картину целиком.

## Блокер при отсутствии Codex

Если `codex-companion.mjs` не найден по 3-tier fallback (`CODEX_COMPANION_NOT_FOUND`), или Codex возвращает `ECONNREFUSED`/ошибку запуска:

- Сообщи Сюзерену: `Codex companion недоступен, intake невозможен. Установи/перезапусти плагин openai-codex и повтори.` Укажи конкретную ошибку из диагностики резолвера.
- **Не переходи** к самостоятельному исполнению. Вся черновая работа (OCR, распаковка архивов, классификация, раскладка) — это работа Codex medium. Claude-main не должен её брать на себя, даже если технически может: юрист специально выделил эту границу ответственности.
- Остановись и жди разрешения проблемы с companion.
