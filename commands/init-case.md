# init-case — Инициализация дела

## Контекст запуска

Команда запускается Сюзереном в той папке, куда он сложил «сырые» материалы от клиента (pdf, docx, сканы, изображения, архивы — всё россыпью в корне). **Текущая рабочая директория (CWD) — это и есть папка дела.** Никаких «создать папку для дела», `mkdir case-name`, `cd`, вопросов «как назвать папку» и «где её создать» быть не должно. Сюзерен уже запустил Claude Code внутри папки дела; всё остальное — работа в CWD.

Скилл сам:

1. Разворачивает в CWD скелет нашей структуры (`.vassal/`, `Входящие документы/`, `Материалы от клиента/`).
2. Сгребает все клиентские файлы из корня CWD в `Входящие документы/`.
3. Запускает intake (переименование, OCR, md-зеркала, раскладка по `Материалы от клиента/`).
4. Извлекает из md-зеркал кандидатов на поля карточки.
5. Опрашивает Сюзерена только по тем полям, которые не смог извлечь сам.

## Последовательность фаз

0. **Preflight** — проверка, что дело ещё не инициализировано.
1. **Scaffold** — создание скелета папок, сбор «сырых» файлов из корня в `Входящие документы/`, заглушки `case.yaml`, `index.yaml`, `history.md`.
2. **Intake** — обработка файлов из `Входящие документы/`.
3. **Extract candidates** — извлечение кандидатов на поля карточки из md-зеркал.
4. **Questionnaire** — опрос Сюзерена через `AskUserQuestion` только по недостающим полям.
5. **Finalize** — запись `case.yaml`, финальная запись в `history.md`, краткая сводка.

## Жёсткие правила

- **Папка дела — это CWD.** Не создавай подпапку для дела. Не спрашивай Сюзерена, где создать дело, как назвать папку или куда положить файлы. Сюзерен уже запустил Claude Code внутри папки дела — `pwd` на момент старта и есть корень дела. Любые `mkdir <case-name>`, `cd <case-name>`, вопросы про имя/расположение папки — грубое нарушение контракта. Все `mkdir` в Фазе 1 — только внутренние (`.vassal/…`, «Входящие документы», «Материалы от клиента»), и всегда в CWD.
- **Файлы клиента лежат в корне CWD**, не в `Входящие документы/`. Сгребай их в `Входящие документы/` из корня (Фаза 1.2). Если в корне CWD пусто — это сигнал «клиент не положил файлы» (обработка — Фаза 2), а не повод перепроверять расположение или спрашивать про путь.
- **Никогда** не спрашивай Сюзерена о полях дела до того, как Фазы 2 и 3 завершены. Смысл скилла — автоматизировать аналитику: сначала сам изучи материалы, потом проси недостающее. Опросник до Фазы 3 — нарушение контракта команды.
- Вопросы задавай только через `AskUserQuestion` (всплывающая форма), не свободным текстом.
- Все человекочитаемые строки в вопросах (тело, опции, description) — **только кириллицей**. Никакого транслита: не пиши «dosudebnyй», «sud», «plaintiff», «parties» — пиши «досудебный», «суд», «истец», «стороны». Если сомневаешься в написании термина — используй короткое нейтральное слово по-русски, но не переключайся на латиницу.
- В текстах вопросов не добавляй от себя поясняющие скобки и комментарии сверх шаблона — только те поля, которые явно разрешены в формате вопроса.

## Фаза 0 — Preflight

1. Зафиксируй, что работаешь в CWD. **Не делай `cd`, не создавай новую папку, не спрашивай про её имя/место.** Всё, что происходит дальше, — в текущей директории. Если хочется «сначала уточнить у Сюзерена, где дело» — это ошибка: дело уже здесь.
2. Проверь, существует ли `.vassal/case.yaml`.
   - Если да — **не перезаписывай**. Сообщи: «Дело уже инициализировано. Обнови карточку вручную или создай новое дело в отдельной папке.» и остановись.
3. Определи `[PLUGIN_ROOT]` по 3-tier fallback из `skills/codex-invocation/SKILL.md`.
4. Если `setup.sh` в этой сессии ещё не запускался — выполни `bash [PLUGIN_ROOT]/scripts/setup.sh` (нужно для Фазы 2: OCR, распаковка архивов, конверсия изображений).

## Фаза 1 — Scaffold

### 1.1. Создание структуры папок

Создай структуру папок. Команда bash (латиница внутри `.vassal/`, кириллица только на уровне корня дела):

```bash
mkdir -p .vassal/{raw,mirrors,analysis,positions,hearings,codex-logs,plans,work}
mkdir -p "Материалы от клиента"
mkdir -p "Входящие документы"
```

Подпапки `.vassal/plans/` и `.vassal/work/` используются контрактом `plan → review → apply` всех файловых скиллов: `plans/` хранит активные планы, которые исполняются apply-фазой (после apply план архивируется в `.vassal/codex-logs/`), `work/` — рабочая область plan-фазы (OCR-тексты, распакованные архивы), обнуляется после apply.

### 1.2. Сбор «сырых» файлов клиента в `Входящие документы/`

До запуска команды Сюзерен сложил документы от клиента прямо в корень папки дела. Перенеси их в `Входящие документы/` одной командой:

```bash
for item in *; do
  [ -e "$item" ] || continue
  case "$item" in
    .vassal|"Материалы от клиента"|"Входящие документы")
      continue ;;
  esac
  mv -- "$item" "Входящие документы/"
done
```

Что сохраняется:
- `.vassal/`, `Материалы от клиента/`, `Входящие документы/` — это наша структура, её не трогаем.
- Скрытые файлы (`.git/`, `.DS_Store`, `.gitignore` и прочие, начинающиеся с точки) — bash-glob `*` по умолчанию их не матчит, они останутся в корне.

Всё остальное (pdf, docx, jpg, zip, подпапки клиента — любое имя без ведущей точки) уедет в `Входящие документы/`. Если в корне было пусто — команда отработает без эффекта, это нормально.

```yaml
    party_id: null

  representative:
    name: null
    short_name: null

    power_of_attorney: null
    address: null
    phone: null
    email: null

    court_fee: null

  pretrial:
    required: null
    required_source: null
    complied: null
    complied_source: null

```

Запиши пустой `.vassal/index.yaml`:

```yaml
version: 2

documents: []

```

Запиши `.vassal/history.md` со стартовой записью:

```markdown
# История действий

## {ГГГГ-ММ-ДД}

### {ЧЧ:ММ} — init-case: скелет создан
- Создана структура папок дела
- Заглушка case.yaml записана (все поля null)
```

## Фаза 2 — Intake входящих документов

1. Посмотри содержимое `Входящие документы/` (Glob). После Фазы 1.2 там должны лежать все материалы клиента.
2. **Если папка пуста** (в корне дела изначально не было файлов) — покажи Сюзерену через `AskUserQuestion`:
   - Вопрос: «В корне папки дела не было файлов. Что дальше?»
   - Options:
     - label: «Я сейчас положу файлы, подожди» — description: «Команда остановится. Положи файлы в корень папки дела (или прямо в `Входящие документы/`) и перезапусти `/vassal-litigator:init-case` или `/vassal-litigator:intake`.»
     - label: «Настроить дело вручную без документов» — description: «Пропустить intake и перейти к опроснику. Материалы дозагрузишь позже через `/vassal-litigator:intake`.»
   - multiSelect: false
   - header: «Intake»
   
   Если выбрано «положу файлы» — остановись с сообщением «Жду файлы. Перезапусти `/vassal-litigator:init-case`, когда положишь их в корень папки дела или во `Входящие документы/`.»
   
   Если выбрано «вручную» — пропусти Фазу 3, перейди к Фазе 4 с пустыми кандидатами.

3. **Если файлы есть** — запусти intake по контракту `plan → review → (revise) → apply → verify`:
   - Прочитай `skills/intake/SKILL.md`.
   - Выполни **Фазу 1 intake (Plan, Codex medium без `--write`)**: собери промпт из `prompts/file-executor-intake-plan.md`, диспатч через companion, дождись завершения, получи путь к markdown-плану в `.vassal/plans/intake-<plan_timestamp>.md`.
   - Выполни **Фазу 2 intake (Review Сюзереном)**: прочитай план целиком и покажи Сюзерену как есть. Собери подтверждение или `revise_feedback`. При правках — перезапусти plan с `revise_feedback` (Фаза 2b intake) столько раз, сколько нужно.
   - Выполни **Фазу 3 intake (Apply, Codex medium `--write`)**: промпт из `prompts/file-executor-intake-apply.md`, диспатч, мониторинг.
   - Выполни **Фазу 4 intake (Verify)**: прочитай отчёт apply, сохрани лог сессии в `.vassal/codex-logs/`, проверь валидность `index.yaml` и то, что `{{plan_path}}` обнулён.
   - Примечание: на этом этапе `case.yaml.case.parties` пустой — Codex plan-фаза сама определит отправителей по OCR-контенту и именам файлов; `Неизвестный` оставит там, где дата/отправитель не извлекаются. Это нормально и не блокирует init-case.
   - Если Codex companion недоступен (`CODEX_COMPANION_NOT_FOUND`, `ECONNREFUSED`) — сообщи Сюзерену то же самое, что написано в `skills/intake/SKILL.md` §«Блокер при отсутствии Codex», и **остановись**. Не переходи к Фазе 3 init-case, не самостоятельно исполняй intake. Сюзерен перезапустит `init-case` после починки companion.

После успешного intake в `.vassal/index.yaml` будут записи документов, в `.vassal/mirrors/` — md-зеркала, в `Материалы от клиента/` — хронологически разложенные файлы.

## Фаза 3 — Extract candidates (Claude-main, read-only)

Цель: собрать из материалов дела кандидатов на поля `case.yaml`, чтобы в Фазе 4 не задавать Сюзерену лишних вопросов.

1. Прочитай `.vassal/index.yaml` — список документов, типы, даты, стороны.
2. Прочитай frontmatter и первые ~2000 символов каждого md-зеркала из `.vassal/mirrors/` (до 10 самых ранних/важных: исковое заявление, договор, претензия, определение суда, доверенность).
3. Извлеки кандидатов:
   - **Стороны**: собери все уникальные наименования из `parties` в `index.yaml` и из полей `parties` в зеркалах. Для каждой стороны определи наиболее вероятную роль (истец / ответчик / третье лицо) по контексту документов (кто подаёт иск, кто получает претензию и т.п.).
   - **Номер дела**: ищи шаблоны `[А-Я]\d+-\d+/\d{4}` (арбитраж) и `\d+-\d+/\d{4}` (СОЮ) в исковом заявлении, определениях суда, повестках.
   - **Суд**: ищи «Арбитражный суд …», «… районный суд …», «… областной суд …» в шапках процессуальных документов.
   - **Судья**: «судья …», «председательствующий …» в определениях и протоколах.
   - **Категория**: из предмета иска (экономические споры / банкротство / административные / общая юрисдикция).
   - **Суть спора**: из предмета исковых требований (1 предложение).
   - **Даты**: `filing_date` из искового заявления, `next_hearing` из определения о назначении заседания.
   - **Сумма иска / госпошлина**: из просительной части искового заявления.
   - **Представитель (наш юрист)**: из доверенности, если есть в материалах.
   - **Досудебный порядок**: дата претензии, если она есть в индексе (`type == "претензия"`).

4. Зафиксируй кандидатов во временной структуре (в голове / в промежуточной переменной) с пометкой уверенности: `high` (встретилось минимум в двух документах), `medium` (встретилось в одном), `low` (спорная эвристика).

Если intake пропущен (Фаза 2 ушла в «вручную») — кандидатов нет, переходи сразу к Фазе 4 с пустыми значениями.

## Фаза 4 — Questionnaire (AskUserQuestion)

Прежде чем входить в Фазу 4, убедись: Фазы 2 и 3 завершены (intake отработал либо явно пропущен через ветку «вручную», кандидаты собраны). Если это не так — вернись назад, не задавай Сюзерену ни одного вопроса о полях дела.

Все вопросы задавай **только** через `AskUserQuestion`. Не пиши свободным текстом «введи номер дела» — пользователь должен увидеть форму.

Весь текст вопросов — только по-русски. Никакого транслита в `question`, `options.label`, `options.description` и `header`. Не добавляй от себя поясняющих скобок и комментариев сверх прописанных ниже шаблонов.

### Q1 — Кто наш клиент (обязательный, всегда первый)

Собери опции из кандидатов-сторон. Правила формирования опций:

- Если из материалов извлечено **≥2 сторон** — каждую сторону выведи как отдельную опцию, в `label` — `«{Название} ({роль})»`, в `description` — краткий контекст из документов.
- Если извлечена **1 сторона** — добавь вторую опцию «Наш клиент не упоминается в текущих материалах» (description: «Напишешь наименование клиента вручную через "Other"»).
- Если сторон извлечено **0** (intake пропущен или ничего не нашлось) — задай вопрос двумя опциями: «Юридическое лицо» и «Физическое лицо», ответ пойдёт через «Other» с вручную вписанным наименованием.

`AskUserQuestion` параметры:
- `question`: «Кто наш клиент в этом деле?»
- `header`: «Клиент»
- `multiSelect`: false
- `options`: сформированные выше (2–4 шт., «Other» добавится автоматически).

После ответа:
- Зафиксируй выбранную сторону как `our_client.party_id`.
- Если ответ через «Other» — создай новую запись в `case.yaml.case.parties` с `party_id: "{role-NN}"` (роль уточни дополнительным вопросом, см. Q2 если нужно).

### Q2…Qn — Недостающие поля (по одному батчу до 4 вопросов)

Формируй батчи `AskUserQuestion` (до 4 вопросов за вызов) **только для полей, где кандидат отсутствует или уверенность `low`**. Поля, где кандидат уверенный (`high` или `medium`), **не спрашивай** — просто подтверди одним общим вопросом в конце (см. Q-final).

Приоритет полей:

1. **Роль каждой стороны** (если не определилась из документов): «Какова роль стороны "{Название}" в деле?» options: `Истец` / `Ответчик` / `Третье лицо` / `Заявитель` (2–4 релевантные для контекста).
2. **Номер дела** (если нет): «Какой номер у дела?» options: `Дело ещё не возбуждено` / `Номер есть (введу через "Other")`. Для номера дела лучше одна опция-плейсхолдер + «Other».
3. **Суд** (если нет): «В каком суде рассматривается дело?» options: 2–3 типовых (`Арбитражный суд` / `Районный суд` / `Мировой суд`) + «Other» для названия.
4. **Судья** (если нет): «Судья известен?» options: `Да, знаю ФИО (через "Other")` / `Нет, ещё не назначен`.
5. **Категория**: options: `Экономические споры` / `Банкротство` / `Административные` / `Общая юрисдикция`.
6. **Даты**: `filing_date`, `next_hearing` — каждая отдельным вопросом с options `Известна (через "Other")` / `Неизвестна`.
7. **Представитель (наш юрист)**: «Представитель клиента — это ты (Сюзерен)?» options: `Да, я` / `Нет, другой юрист (через "Other")`.
8. **Суммы**: `claim` — options: `Не применимо (не денежный спор)` / `Сумма известна (через "Other")`.
9. **Досудебный порядок**: если в материалах нет претензии — спроси «Направлялась ли претензия?» options: `Да` / `Нет` / `Не требуется по закону`.

Группируй связанные вопросы в один вызов `AskUserQuestion` (до 4 вопросов). Не задавай вопросы, ответ на которые уже есть в кандидатах с высокой уверенностью.

### Q-final — Подтверждение извлечённых полей

Если в Фазе 3 были извлечены поля с уверенностью `high` или `medium` — покажи их Сюзерену одним вопросом.

Шаблон `question` (собирай построчно, строго в этом порядке, каждое поле на своей строке, формат `Название: значение`; никаких дополнительных полей, комментариев в скобках и эмодзи):

```
Я извлёк из материалов следующие данные. Всё верно?

Номер дела: {номер или «не извлечён»}
Суд: {название или «не извлечён»}
Судья: {ФИО или «не извлечён»}
Категория: {категория или «не извлечена»}
Стороны: {для каждой стороны — «Роль: Название»; разделитель — « · »}
Наш клиент: {Название (роль)}
Суть спора: {одно предложение из искового или «не извлечена»}
Сумма иска: {число с указанием валюты или «не извлечена»}
Дата претензии: {ГГГГ-ММ-ДД или «не извлечена»}
```

Правила формирования:

- Если поля нет среди кандидатов `high`/`medium` — не включай его в тело вопроса вообще (не пиши «не извлечён» для пропущенных полей, пиши только для тех, которые пытался извлечь).
- Не добавляй поля, которых нет в шаблоне выше (например, не пиши «Третье лицо: …» отдельной строкой — третьи лица идут в «Стороны»).
- Не пиши пояснительных скобок типа «(да — досудебный порядок соблюдён)». Факты, не попавшие в шаблон, задавай отдельным вопросом в Q2…Qn.
- Валюта, даты, номера — пиши буквально как в документах (кириллицей, с ёлочками, без транслита).

Параметры `AskUserQuestion`:

- `header`: «Проверка»
- `multiSelect`: false
- `options`:
  - label: «Всё верно»
