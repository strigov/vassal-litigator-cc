## Goal

Плагин `vassal-litigator-cc` сейчас выполняет всю файловую рутину (распаковка архивов, предварительное OCR, сканирование `index.yaml` vs filesystem, классификация качества OCR, копирование исходников, генерация md-зеркал, обновление `index.yaml`) внутри LLM-промптов Codex medium и Claude-main: каждый запуск тратит токены на одинаковые механические шаги, плодит вариативность исполнения и затрудняет аудит. Эталонный плагин `vassal-litigator` уже имеет все нужные детерминированные утилиты (`prepare_intake_workdir.py`, `scan_case_state.py`, `classify_ocr_quality.py`, `apply_intake_plan.py`) с покрытием unit-тестами; они проверены в проде и переносимы.

Цель — перенести в `vassal-litigator-cc` **только те** утилиты, заменяющие чисто механическую работу LLM (без потери LLM-функций, требующих смысловых решений: классификация дат/типов/комплектов из содержимого, ревью, ревайз). После переноса plan-фаза остаётся LLM-задачей (Codex читает OCR, классифицирует, строит хронологию), но apply-фаза становится исполнением машинного плана, а update-index выводит детерминированный diff в preview без LLM-сканирования файловой системы.

Вместо этого patch использует **longest-prefix parsing**: список известных префиксов `KNOWN_OP_PREFIXES = ["add-evidence-", "add-opponent-", "intake-"]` (порядок важен — `add-evidence-` и `add-opponent-` ДО `intake-`, иначе наивный split дал бы для `add-evidence-2026-...` префикс `add`); функция `_resolve_operation_token(batch: str) -> str` перебирает префиксы по длине (longest first), при первом match возвращает соответствующую history-метку через маппинг `{"intake-" → "intake apply", "add-evidence-" → "add-evidence apply", "add-opponent-" → "add-opponent apply"}`; если ни один префикс не подошёл — ApplyError с явным сообщением, перечисляющим допустимые префиксы. Patch локализован в одной функции `_history_line` + новой helper `_resolve_operation_token`; unit-тест `tests/unit/test_apply_intake_plan.py::test_history_line_per_skill` (плагин-локальная фикстура поверх перенесённого reference-теста) покрывает все три префикса с **реальными basenames** (`intake-2026-05-05-1430`, `add-evidence-2026-05-05-1430`, `add-opponent-2026-05-05-1430`) + negative для неизвестного префикса (`unknown-2026-05-05-1430`) + negative-edge для `add-2026-05-05-1430` (короче чем `add-evidence-`/`add-opponent-`, не должен матчить `add-evidence apply`).

```

{
  "work_dir": "<abs path>",

    {
      "source_path": "<abs path в inbox или в <work_dir>/<archive_stem>/...>",

      "needs_image_to_pdf": <bool>,

      "ocr_artifact_path": "<abs path в <work_dir>/extracted/<name>.txt>" | null
    },
    ...
  ],
  "archives_unpacked": [{"archive": "<abs>", "extracted_to": "<abs>"}],
  "unsupported": [{"archive": "<abs>", "reason": "..."}]

exit 0 при успехе; ненулевой код + stderr при сбое.

python3 apply_intake_plan.py <case_root> --plan-yaml <abs path в <case_root>/.vassal/plans/> [--dry-run] [--force]
→ stdout JSON (фактический контракт reference-скрипта):
{
  "applied": <bool>,                 # true = реально применили; false = dry-run
  "batch": "<batch id>",
  "added_doc_ids": ["doc-007", ...],
  "converted_images": <int>,
  "bundle_count": <int>,
  "orphan_count": <int>,             # = items[] с date == null
  "raw_batch_path": "<abs path к raw_dest>",
  "history_line": "<строка для .vassal/history.md>",
  "state_file": ".vassal/plans/<batch>-apply-state.json",
  "cleanup_errors": ["..."]
}
exit 0 при успехе. При любой валидационной ошибке скрипт пишет сообщение в stderr и выходит ненулевым кодом — JSON в stdout НЕ гарантирован при ошибке.
```

Дополнительные жёсткие требования reference-скрипта (которые prompts/skills обязаны соблюдать):
- `--plan-yaml` ДОЛЖЕН быть абсолютным путём строго внутри `<case_root>/.vassal/plans/`. Любое имя допускается, но конвенцию плагин-локального именования см. ниже.
- При успешном `apply` (НЕ dry-run) скрипт **сам**: (а) удаляет staging `<case_root>/.vassal/tmp/apply-<batch>/`; (б) удаляет `validated["work_dir"]` (целиком); (в) удаляет sibling `<plan_path>.with_suffix(".md")` и `<plan_path>` (если расширение `.yaml`); (г) удаляет `state_file`. Поэтому prompt apply-фазы НЕ должен сам чистить `work_dir` или удалять plan-файлы — иначе будут двойные FileNotFound.
- Из-за пункта (в) имя `<markdown_plan>.machine.yaml` **сломает** связку: `with_suffix(".md")` превратит `intake-...-1430.md.machine.yaml` в `intake-...-1430.md.machine.md` и снесёт уже не markdown-плана, а посторонний/несуществующий путь, оставив исходный markdown сиротой. Конвенция должна быть **`<plan_basename>.yaml`** рядом с `<plan_basename>.md` (например, `intake-2026-05-05-1430.md` + `intake-2026-05-05-1430.yaml`) — тогда `with_suffix(".md")` корректно укажет на парный markdown.
- Plan-yaml и markdown-план попадают в `[CASE_ROOT]/.vassal/codex-logs/<plan_basename>.{md,yaml}` (абсолютный путь под `.vassal/`, единый для всех скиллов плагина-CC) через apply-prompt **до** запуска скрипта (потому что после apply файлов уже не будет). Apply-prompt сам делает `mkdir -p "[CASE_ROOT]/.vassal/codex-logs"` перед `cp`.
- При повторном запуске того же `--plan-yaml` (если `state_file` ещё на диске) скрипт продолжит promotion из staging, но при чистом состоянии «уже применённый план» вызовет `doc_id already exists in index` → ApplyError → ненулевой exit. Это «no-op + warning» на уровне prompts только в строгом смысле «второй apply рапортует BLOCKED, существующий index не повреждён». Полный idempotent rerun через `--force` (флаг no-op в reference) пока недоступен.

### Машинный план *(Ф2, shared/plan-schema.yaml)*

Контракт machine-plan **фиксируется фактическим валидатором `_validate_plan` в `apply_intake_plan.py`** — переименование/добавление/удаление полей в `shared/plan-schema.yaml` без правок скрипта приведёт к BLOCKED. Поэтому `shared/plan-schema.yaml` оформляется как информативное JSON-Schema-описание реального контракта (для prompt-кодогенерации), а не как независимый источник истины.

**Batch identity invariant (BLOCKING #2 текущего ревью).** Существующие skill-конвенции этого плагина и аналог в reference-плагине используют `batch_name` уровня скилла-в-день: `intake-YYYY-MM-DD`/`evidence-YYYY-MM-DD`/`opponent-YYYY-MM-DD`. `apply_intake_plan.py` ключует по `batch` всё критичное состояние: `state_file = .vassal/plans/<batch>-apply-state.json`, staging-каталог `.vassal/tmp/apply-<batch>/`, `history.md`-строку, `origin.batch` каждого item в индексе и (по нашей конвенции) `raw_dest = .vassal/raw/<batch>`. Если два intake/add-evidence/add-opponent запускаются в один день (что нормально — рассыпанные доставки в течение дня, повторный intake после revise, отдельные комплекты от разных контрагентов), date-only `batch` коллидирует: stale `state_file` от предыдущего apply будет переиспользован при новом apply, raw-каталог с тем же именем перезапишется, `history.md` получит дубль-строки, `origin.batch` потеряет различимость. Поэтому в плагине-CC `batch` machine-plan — это **уникальный per-plan basename**, идентичный `<plan_basename>` markdown-плана: `<skill>-YYYY-MM-DD-HHMM` (минута запуска plan-фазы Codex), например `intake-2026-05-05-1430`. Гарантии:
- plan-фаза каждого скилла (Ф2 step 3) при генерации `<plan_basename>.md` и `<plan_basename>.yaml` использует **один и тот же** basename из текущей даты-времени локального часового пояса; этот же basename подставляется в верхнеуровневое поле `batch` machine-plan;
- `work_dir` и `raw_dest` plan-фаза формирует по правилу `<case_root>/.vassal/work/<batch>` и `<case_root>/.vassal/raw/<batch>` соответственно — это сохраняет коллизионную безопасность state/staging/raw в один присест;
- collision check (часть `validate_machine_plan.py`, доп. инвариант) — **режим-зависимый (BLOCKING #2)**. Общие проверки, обязательные в обоих режимах: (а) `Path(plan_yaml).stem != plan["batch"]` (mismatch между именем файла и полем `batch`) → FAIL; **(а')** `realpath(plan["work_dir"]) != realpath(<case_root>/.vassal/work/<batch>)` → FAIL (BLOCKING #3: явная проверка, что `work_dir` действительно привязан к batch — без неё валидный план может удалить чужой `work_dir` через `_final_cleanup`, который делает `shutil.rmtree(validated["work_dir"])` без guard'а на принадлежность batch'у); **(а'')** `realpath(plan["raw_dest"]) != realpath(<case_root>/.vassal/raw/<batch>)` → FAIL (BLOCKING #3: без неё план может писать raw в произвольный каталог под `.vassal/raw/`, e.g. перезатирая raw чужого batch). Режим-зависимые проверки:
  - `--mode plan` (запуск из plan-фазы, ДО создания state_file/raw_dest): (б) `<case_root>/.vassal/raw/<batch>` существует и НЕ пуст → FAIL (ранее применённый план под тем же basename); (в) `<case_root>/.vassal/plans/<batch>-apply-state.json` существует → FAIL (stale state от прерванного предыдущего apply должен быть либо доведён до конца, либо удалён вручную после ручного аудита; в plan-фазе не должно быть никакого state).
  - `--mode apply` (запуск из apply-фазы, прямо перед `apply_intake_plan.py`): (б') `state_file` существует И валиден (JSON-загружается, содержит `plan_yaml`, `batch` И `plan_sha256`-поля) И `state["plan_yaml"] == realpath(<plan_yaml>)` И `state["batch"] == plan["batch"]` И **`state["plan_sha256"] == sha256(<plan_yaml> bytes)`** (BLOCKING #3, см. Resume content guard ниже) → **PASS** (это легитимный resume того же плана с неизменённым содержимым); если `state_file` существует, но `plan_yaml`/`batch` не совпадают — FAIL (state от другого плана с тем же basename); если `state_file` существует, но `plan_sha256` отсутствует или не совпадает — FAIL (plan-yaml был отредактирован после прерванного apply, drift); если `state_file` существует, но JSON битый — FAIL (deferred manual recovery). (в') `raw_dest` непуст → PASS, **iff** `state_file` присутствует и валидно ссылается на тот же plan_yaml И `plan_sha256` совпадает (это означает, что предыдущий apply начал promotion, отвалился и сейчас ресюмится `_complete_after_state` с тем же содержимым плана); если `raw_dest` непуст БЕЗ соответствующего валидного `state_file` → FAIL.

**Resume content guard (BLOCKING #3).** `state["plan_yaml"]` (путь) и `state["batch"]` сами по себе НЕ защищают от ситуации, когда между прерванным `apply_intake_plan.py` и повторным запуском пользователь (или Codex revise) подправляет содержимое YAML — например, меняет `target_file`, добавляет/удаляет item, корректирует bundle membership. После такой правки validate-wrapper по старой версии «совпали ли путь и batch» отдал бы exit 0, `apply_intake_plan.py` через ветку `existing_state is not None` → `_complete_after_state` промотировал бы УЖЕ-staged артефакты (которые отражают **старый** plan), но сформировал бы report и cleanup на основе **нового** plan-yaml: history.md получил бы строку с актуальным `len(items)`, а в `index.yaml` попали бы doc-ID из старого staging — критический рассинхрон. Защита:
- (1) `apply_intake_plan.py` (plugin-local patch ко всем местам, где скрипт пишет state — `_save_state`/`_safe_write_json` для `state_file`): при сохранении state-файла дополнительно класть в JSON поле `plan_sha256` со sha256-хешем `<plan_yaml>` bytes на момент первого `_build_staging`. Reference-скрипт уже пишет `plan_yaml` и `batch`; добавляется ровно одно поле. Patch локализован в одной функции; unit-тест `tests/unit/test_apply_intake_plan.py::test_state_file_carries_plan_sha256` (плагин-локальный) проверяет наличие поля и его соответствие реальному hash.
- (2) `validate_machine_plan.py` в `--mode apply` при наличии state-файла читает `state["plan_sha256"]` и сравнивает с sha256 текущего `<plan_yaml>` (`hashlib.sha256(open(plan_yaml,'rb').read())`). Несовпадение → FAIL с сообщением `resume drift: plan content changed since previous apply (state sha256=<x>, current sha256=<y>); restore previous YAML or remove state_file to start fresh`.
- (3) **Codex-log backup preservation на resume.** Текущая редакция apply-prompt безусловно делает `cp .vassal/plans/<plan_basename>.{md,yaml} .vassal/codex-logs/` ДО запуска скрипта (см. Files Ф2 → apply-prompt шаг 1). Если предыдущий apply прервался уже после такого backup, повторный apply с **отредактированным** YAML перезатрёт codex-log backup новой (drift) копией ДО валидатора и потеряет ground truth. Patch apply-prompt: шаг 1 переписывается в conditional cp — `if [ -f "[CASE_ROOT]/.vassal/codex-logs/<plan_basename>.yaml" ] && [ -f "[CASE_ROOT]/.vassal/plans/<batch>-apply-state.json" ]; then echo "preserving existing codex-log backup (resume path)"; else mkdir -p ... && cp ...; fi`. Тем самым на resume backup живёт нетронутым; валидатор в `--mode apply` дополнительно сравнивает sha256 текущего `<plan_yaml>` с sha256 backup-копии из codex-logs — несовпадение → FAIL (та же причина: plan был отредактирован между прерванным apply и resume). На fresh apply (state отсутствует) cp выполняется как раньше.
- (4) Negative test `tests/unit/test_validate_machine_plan.py::test_resume_drift_blocked` (BLOCKING #3): фикстура — валидный prepared resume (state_file присутствует, raw_dest staged, codex-log backup существует), затем содержимое `.vassal/plans/<plan_basename>.yaml` модифицируется (добавляется лишний whitespace или меняется `title` одного item — sha256 расходится) → `validate_machine_plan.py --mode apply` exit non-zero с упоминанием `resume drift`. Парный positive — без модификации YAML wrapper exit 0, последующий `apply_intake_plan.py` доводит promotion, exit 0.
- эти правила точно отражают штатный resume-флоу `apply_intake_plan.py` (см. ветку `existing_state is not None` в `main()`, строки 1207–1221 reference-скрипта): скрипт сам корректно обрабатывает повторный запуск с того же plan-yaml при сохранённом `state_file`, поэтому wrapper-валидатор в `--mode apply` обязан НЕ блокировать этот путь;
- SKILL.md контракты `intake`/`add-evidence`/`add-opponent` обновляются (Ф2 step 7): любое упоминание `batch_name = <skill>-YYYY-MM-DD` заменяется на `batch = <plan_basename> = <skill>-YYYY-MM-DD-HHMM`; в Verify-секции добавляется явный assertion `plan["batch"] == <plan_basename>` (читать из бэкап-копии в `.vassal/codex-logs/<plan_basename>.yaml`);
- contract-тест `tests/unit/test_validate_machine_plan.py::test_batch_identity` — фикстуры: (a) `batch != plan_basename` → exit non-zero в обоих режимах; **(a')** `work_dir != <case_root>/.vassal/work/<batch>` → exit non-zero в обоих режимах (BLOCKING #3); **(a'')** `raw_dest != <case_root>/.vassal/raw/<batch>` → exit non-zero в обоих режимах (BLOCKING #3); (b) `raw_dest` существует и непуст без `state_file` → exit non-zero в обоих режимах; (b') `raw_dest` существует и непуст ПЛЮС валидный `state_file` ссылается на тот же plan_yaml → `--mode plan` exit non-zero, `--mode apply` **exit 0** (легитимный resume); (c) sibling `<batch>-apply-state.json` существует с матчингом `plan_yaml`/`batch` → `--mode plan` exit non-zero, `--mode apply` **exit 0**; (c') sibling `<batch>-apply-state.json` существует с НЕсовпадающим `plan_yaml` или `batch` → exit non-zero в обоих режимах; (c'') sibling `<batch>-apply-state.json` существует, но JSON битый → exit non-zero в обоих режимах; (d) согласованный `batch == plan_basename` + чистый `raw_dest`/`state_file` + `work_dir`/`raw_dest` под `<batch>` → exit 0 в обоих режимах;
- smoke `tests/smoke/test-intake.sh` (и аналоги): два apply-захода в один день должны давать **разные** `<plan_basename>` (формат включает `HHMM`), независимые `raw_dest`, независимые `state_file`, две отдельные строки в `history.md`. Тест явно эмулирует два захода с разницей в одну минуту.

Важно для BLOCKING #3 (ранее ожидавшиеся метаданные): `apply_intake_plan.py` НЕ принимает поля `parties`, `summary_draft`, `signatures_present`, `seal_present`, `complete`, `quality` и любые другие plugin-extras в frontmatter/index. Расширение этого контракта потребовало бы патча скрипта и его тестов — это вне scope текущего переноса. Поэтому в Ф2 решение принимается **сужать контракт** в plagin-CC до фактически поддерживаемых скриптом полей:
- `parties` — НЕ записывается в `index.yaml`/mirror через скрипт; если plugin-CC сегодня требует его в frontmatter, то либо (а) удаляем требование из prompts/skills/`shared/mirror-template.md` (предпочтительный путь, согласованный с reference-плагином), либо (б) добавляем post-step в apply-prompt, который дополняет frontmatter после скрипта. В Ф2 выбираем (а).
- `summary_draft`, печать/подпись/комплектность, `quality` (как явное поле): плагин-CC удаляет из всех intake/add-evidence/add-opponent prompts/skills. Финальные `ocr_quality`/`ocr_quality_reason`/`needs_manual_review` пишутся скриптом **только в `index.yaml`** (через `_build_index_payload`, который вызывает `classify_ocr_quality.classify` и кладёт три поля в каждую запись `documents[]`). В **mirror frontmatter** этих полей НЕТ: фактический `_build_mirror_text` ограничивается набором `id`/`source`/`type`/`title`/`date`/`pages`/`extraction_method`/`extraction_model`/`extraction_date`/`confidence`/`ocr_reattempted`/`extraction_methods_mixed?` (см. строки 391–413 reference-скрипта). Любая prompt/skill инструкция «убедись, что `ocr_quality` есть в frontmatter зеркала» неверна и должна быть удалена. Если плагин-CC требует видеть `ocr_quality` в зеркале — это отдельный план: расширить `_build_mirror_text` и его тесты в reference, ИЛИ добавить post-step в apply-prompt после скрипта, который дополняет mirror frontmatter (но это ломает идемпотентность и не рекомендуется).
- mirror-frontmatter, рендерится скриптом, содержит ровно: `id`, `source` (relative path к target_file), `type`, `title`, `date`, `pages`, `extraction_method`, `extraction_model: "haiku"`, `extraction_date`, `confidence`, `ocr_reattempted: false`, опционально `extraction_methods_mixed: true`. `shared/mirror-template.md` НЕ удаляется (см. Files Ф2): остаётся как канонический шаблон для update-index apply (LLM-driven), приводится в соответствие с реальным форматом `_build_mirror_text`, чтобы зеркала, созданные обоими путями, были одинаковы. Когда/если update-index apply будет переведён на скрипт — шаблон можно удалять.

`apply_intake_plan.py` НЕ выполняет strict unknown-key rejection: лишние ключи в верхнем уровне или в `items[]` молча игнорируются (`_validate_plan` читает только известные ключи). Таким образом контракт «schema_version/skill/markdown_plan/next_id_after/archives_kept_as_raw_only/unsupported_archives/parties/doc_type/...», описанный в первой версии плана, технически не сломает apply (поля просто не используются), но и не даст никакой гарантии. Поэтому plan-фаза НЕ выводит эти поля; smoke/contract-тесты проверяют отсутствие лишних ключей.

**Doc-ID contiguity guard (BLOCKING #1).** `_validate_plan` проверяет `next_id_start == index.next_id` и уникальность каждого `doc_id` в формате `doc-NNN`, но **НЕ проверяет**, что числовые суффиксы в `items[].doc_id` образуют непрерывный диапазон `[next_id_start, next_id_start + len(items) − 1]`. После apply скрипт устанавливает `next_id = index.next_id + len(items)` (см. `_build_index_payload`, строка 1003 reference), независимо от того, какие конкретно doc-ID были использованы. Следствие: plan с `next_id_start=47` и `items=[doc-999]` пройдёт dry-run и apply; в индекс попадёт `doc-999`, а `next_id` станет `48`. При следующем intake plan-фаза получит `next_id_start=48`, и пользователь до конца жизни кейса будет иметь дыру `doc-48..doc-998` плюс уже занятый `doc-999`, который рано или поздно столкнётся с автоинкрементом. Скрипт вызывается plan-фазой ПЕРЕД `apply_intake_plan.py --dry-run`: если обёртка падает, plan-фаза BLOCKED, Codex обязан перегенерировать.

**Skipped vs cleanup_set safety (BLOCKING #5, дополненный после R3 для grouped_inputs и pre-convert).** `apply_intake_plan._validate_plan` принимает `skipped[]` (валидирует и кладёт в state.json), но НЕ копирует пути из `skipped[]` ни в `raw_dest`, ни в `items[]`, ни в индекс — это чистый «логирующий» список. При этом `cleanup_set[]` после успешного apply удаляет указанные пути из `Входящие документы/`. Если plan-фаза включит skipped-файл одновременно в `cleanup_set`, файл будет удалён без какой-либо копии и пользователь потеряет данные при повторной отправке/повторной обработке. Поэтому plan-фаза каждого скилла обязана соблюдать инвариант:
- путь, попавший в `skipped[]` (например, password-protected архив без `raw_only`-ветки, или нечитаемый файл, или дубликат уже существующего `doc-NNN`), **НЕ может одновременно попасть в `cleanup_set[]`**;
- если файл нужно убрать из inbox, plan-фаза обязана выбрать ОДИН из безопасных путей: (а) включить в `raw_only[]` (тогда apply скопирует архив в `raw_dest` до удаления из inbox); (б) включить как item с `target_file` в `case_root` через одиночный `source_path` или **`grouped_inputs[]`** (тогда apply копирует каждый исходник в raw + кладёт в дерево дела); (в) оставить в inbox (НЕ добавлять в `cleanup_set`) и зафиксировать `skipped[].reason` для последующего ручного решения Сюзерена;
- **safety-инвариант для `cleanup_set[]`** (расширенный после R3 и уточнённый после R4 для архивов): для КАЖДОГО пути `p ∈ cleanup_set` хотя бы одно из условий должно выполняться: (i) существует item с `item.source_path == p` (т.е. `p` — сам исходник, который apply скопирует в `raw_dest` через `_build_staging`); (ii) существует item с `p ∈ item.grouped_inputs` (тот же резон — каждый элемент `grouped_inputs` копируется в `raw_dest`); (iii) **архив `p` зарегистрирован в `raw_only[].archive_src == p`** — это единственный путь, который гарантирует bit-копию исходного архива в `raw_dest` до удаления из inbox. **`item.archive_src == p` сам по себе НЕ является достаточным условием для удаления `p`**: поле `archive_src` на уровне item только помечает провенанс распакованных файлов (для `origin.archive_src` в индексе), но `apply_intake_plan._build_staging` копирует в `raw_dest` ТОЛЬКО `source_path`/`grouped_inputs` (распакованные файлы из `work_dir`), а сам архив из `source_inbox` не трогает. Если plan-фаза включит архив в `cleanup_set` лишь потому, что какой-то item ссылается на него через `item.archive_src`, после apply архив будет физически удалён из inbox без какой-либо bit-копии — оригинал утрачен. Поэтому для удаления архива из inbox обязательна **отдельная** запись `raw_only[].archive_src == p` (либо сам архив должен быть полноценным `item.source_path`, что бессмысленно для распакованных архивов и применимо только к monolithic-архивам без распаковки); (iv) для pre-converted одиночного изображения (см. Single-image constraint, путь (b)) — оригинальное JPG/PNG в `source_inbox` **обязано** быть зарегистрировано в `raw_only[].archive_src` ИЛИ продублировано как отдельный item, иначе попадание этого исходника в `cleanup_set` запрещено. Дополнительно: пересечение `cleanup_set ∩ {skipped[].path}` ДОЛЖНО быть пусто;
- contract-тест `tests/unit/test_machine_plan_contract.py` обязан включать assertion-фикстуры: одиночный source_path в cleanup_set (PASS), grouped_inputs-набор в cleanup_set (PASS), skipped-путь в cleanup_set (FAIL до apply через wrapper-валидатор), pre-converted PDF из work_dir в cleanup_set (FAIL — work_dir вне source_inbox, скрипт сам отвергнет; но оригинальный JPG из inbox должен идти в raw_only ИЛИ быть отдельным item, иначе wrapper-валидатор тоже FAIL); **negative fixture "archive_src in cleanup_set without raw_only" — plan содержит item с `item.archive_src=<inbox>/archive.zip` и распакованными `grouped_inputs` из `work_dir`, при этом `<inbox>/archive.zip` указан в `cleanup_set` БЕЗ соответствующей `raw_only[].archive_src` записи: wrapper-валидатор обязан вернуть ненулевой exit с сообщением о потере bit-копии архива; позитивный парный case — та же фикстура + добавленная `raw_only[{archive_src: <inbox>/archive.zip, raw_dest_name: archive.zip}]` запись: wrapper exit 0, smoke после apply подтверждает наличие архива в `raw_dest/archive.zip`**;
- smoke `tests/smoke/test-intake.sh`: после apply проверить, что все файлы из `skipped[].path` (отрезок до apply, читать из плана через `[CASE_ROOT]/.vassal/codex-logs/<plan_basename>.yaml`) либо всё ещё в inbox (если не были в cleanup_set), либо лежат в `raw_dest`/`target_file` (если были перемещены через item/raw_only); дополнительно — что для каждого pre-converted single-image кейса оригинальное изображение присутствует в `raw_dest` (через `raw_only` или отдельный item).

**Single-image constraint (BLOCKING #3, фиксируется без патча скрипта).** `prepare_intake_workdir.py` помечает любые JPG/PNG в inbox флагом `needs_image_to_pdf=true`, но `apply_intake_plan.py._validate_plan` принимает `convert_image_to_pdf=true` ТОЛЬКО при `grouped_inputs` длины ≥ 2 (одиночный `source_path` + `convert_image_to_pdf=true` сразу даёт `ApplyError("convert_image_to_pdf requires grouped_inputs")`). Дополнительно: `image_to_pdf.py` создаёт **одну страницу PDF на каждый входной файл** — поэтому «продублировать одиночное изображение как `grouped_inputs=[image, image]`» НЕЛЬЗЯ: получится двухстраничный PDF, в котором оригинальная одностраничная картинка превращается в дубль, и метаданные `pages`/OCR удваиваются. Это коррумпирует целевой документ. Чтобы не блокировать intake/add-evidence/add-opponent при одиночных JPG/PNG и при этом не плодить дубль-страниц, plan-фаза каждого скилла обязана:
- если в `files[]` есть **несколько** изображений одного логического документа (типовой кейс — фото договора по страницам), сгруппировать их в один `items[]` с `grouped_inputs[]` длины ≥ 2 и `convert_image_to_pdf=true`. `grouped_inputs[]` зарезервирован **только** для реальных многостраничных наборов изображений; дублирование одной картинки запрещено;
- если изображение действительно одиночное (`needs_image_to_pdf=true` и нет других страниц), **единственный допустимый путь** — pre-convert плагином-локально через `python3 "[PLUGIN_ROOT]/scripts/image_to_pdf.py" --in "<inbox>/<image>" --out "<work_dir>/<name>.pdf"`, затем в machine-plan указать `source_path=<work_dir>/<name>.pdf` + `convert_image_to_pdf=false`, **обязательно** добавить запись в `raw_only[]` с `archive_src=<inbox>/<image>` и `raw_dest_name=<safe basename исходного изображения>`. Это сохраняет инвариант «raw_dest содержит оригинальный JPG/PNG, а не только PDF-производную»: `_build_staging` для `raw_only[]` копирует исходник в `raw_dest` напрямую (строка 845 reference). Без `raw_only`-копии оригинальное изображение будет утрачено при cleanup_set (см. safety-инвариант пункт (iv) в Skipped vs cleanup_set). Pre-convert НЕ должен записывать PDF поверх исходника в inbox — только в `work_dir`. `image_to_pdf.py` переносится вместе с `apply_intake_plan.py` в Ф2 step 1.
- contract-тест `tests/unit/test_machine_plan_contract.py` обязан включать ТРИ fixture-вариации:
  - (a) **истинно многостраничное** изображение (≥2 разных файла) → `grouped_inputs=[img1, img2, ...]` + `convert_image_to_pdf=true` → smoke проверяет, что после apply (i) `target_file` — многостраничный PDF c `pages == len(grouped_inputs)`, (ii) каждый оригинал лежит в `raw_dest` через `_build_staging` `grouped_inputs`-ветку;
  - (b) **истинно одиночное** изображение → pre-converted PDF в `work_dir` (`source_path=<work_dir>/<name>.pdf`, `convert_image_to_pdf=false`) + `raw_only[]` с `archive_src=<inbox>/<image>` → smoke проверяет, что после apply (i) `target_file` — одностраничный PDF (`pages == 1`), (ii) оригинальный JPG/PNG лежит в `raw_dest/<raw_dest_name>` (через `raw_only`-ветку), (iii) `raw_dest` НЕ содержит дубль PDF под именем исходника, (iv) OCR-метаданные не удвоены;
  - (c) **negative**: одиночное изображение → `grouped_inputs=[image, image]` (дубль того же файла) + `convert_image_to_pdf=true` → wrapper-валидатор FAIL до apply (новый sub-инвариант: все элементы `grouped_inputs[]` должны быть **разными** resolved real paths);
  - (d) **negative**: одиночное pre-converted PDF в work_dir БЕЗ `raw_only`-копии оригинала → wrapper-валидатор FAIL до apply.
  Это формализует workflow и предотвращает регрессию «plan-фаза снова дублирует одно изображение в grouped_inputs» И «plan-фаза pre-converted без raw_only-копии». Альтернативный путь «патчить `_validate_plan` под одиночные изображения» вне scope текущего переноса.

**Raw destination collision guard (R6 BLOCKING #2).** `_validate_plan` в `apply_intake_plan.py` НЕ проверяет, что разные источники не записываются в один и тот же путь внутри `raw_dest`. `_build_staging` (строки 843–882 reference-скрипта) формирует final raw destination ПО-РАЗНОМУ для двух путей: (i) для `raw_only[i]` final dest — `raw_dest / raw_only[i].raw_dest_name` (плоский basename, строка 845 reference); (ii) для каждого `item.source_path` и `item.grouped_inputs[k]` — final dest строится через `rel_path = resolved_source.relative_to(source_root)`, где `source_root = source_inbox` если source лежит в inbox, иначе `work_dir`; затем **final = `raw_dest / rel_path`** (строка 879 reference: `staged_dst = plan["raw_dest"] / rel_path`). **Поле `item.archive_src` на final raw dest НЕ влияет**: префикс `archive_src.name` фигурирует ТОЛЬКО во временном staging-бакете (`staging_root / "raw" / archive_src.name / rel_path`, строки 863–864 + 876), но при promotion из staging в `raw_dest` используется чистый `rel_path` без префикса. Это означает, что валидный план может иметь `<inbox>/sub/contract.pdf` (final raw dest = `raw_dest/sub/contract.pdf`) и одновременно `raw_only[{archive_src:..., raw_dest_name:"contract.pdf"}]` (final raw dest = `raw_dest/contract.pdf`) — это РАЗНЫЕ пути и коллизии нет. Простой basename-guard здесь даст ложный BLOCKED. Поэтому `validate_machine_plan.py` обязан считать **полный** raw destination по тем же правилам, что и `_build_staging`, и падать только если один и тот же final destination имеет разные resolved sources:
- собрать список пар `(final_raw_dest_path, resolved_source_realpath)` ровно по правилам `_build_staging`:
  - для каждого `raw_only[i]`: `final = raw_dest / raw_only[i].raw_dest_name`, `source = realpath(raw_only[i].archive_src)`;
  - для каждого `item.source_path`: `source_root = source_inbox` если `source_path` внутри `source_inbox`, иначе `work_dir`; `rel_path = realpath(source_path).relative_to(realpath(source_root))`; **`final = raw_dest / rel_path`** (без префикса `archive_src.name` — он живёт только во временном staging-бакете и не доезжает до final dest); `source = realpath(source_path)`;
  - для каждого `item.grouped_inputs[k]`: то же по правилу `source_path` (источник может быть в `source_inbox` ИЛИ в `work_dir`); поле `item.archive_src` НЕ участвует в вычислении final dest;
- сгруппировать пары по `final_raw_dest_path` (как строка после `os.path.normpath`); для каждой группы проверить, что все `source_realpath` в группе равны (т.е. это физически один и тот же файл, например, дублирующая ссылка на один source). Если хотя бы одна группа содержит ≥2 РАЗНЫХ `source_realpath` — FAIL: `raw destination collision: <final_dest> ← {<src1>, <src2>}`;
- частный случай (из R6): два `raw_only[]` с одинаковым `raw_dest_name` и разными `archive_src` → их `final` совпадает, resolved sources разные → FAIL;
- частный случай (R7 ложный позитив, который guard НЕ должен ловить): `raw_only.raw_dest_name="contract.pdf"` + `item.source_path=<inbox>/sub/contract.pdf` (с `archive_src=null`) → finals разные (`raw_dest/contract.pdf` vs `raw_dest/sub/contract.pdf`) → PASS;
- частный случай (реальная коллизия после relative_to): `item_a.source_path=<inbox>/contract.pdf` (`archive_src=null`, final = `raw_dest/contract.pdf`) + `raw_only.raw_dest_name="contract.pdf"` с другим `archive_src` → finals совпадают, resolved sources разные → FAIL;
- **частный случай (реальная коллизия, которую basename- или archive_src-prefix-guard пропустил бы): item из `work_dir` с `archive_src != null`** — `item_a.source_path=<work_dir>/extracted/foo/bar.pdf` (`archive_src=<inbox>/archive.zip`, `source_root=work_dir`, `rel_path=extracted/foo/bar.pdf`, final = `raw_dest/extracted/foo/bar.pdf`) **+** другой источник, дающий тот же final, например `item_b.grouped_inputs=[<work_dir>/extracted/foo/bar.pdf, ...]` под другим item с другим `archive_src=<inbox>/other.zip` (тот же физический файл из общего work_dir, но если по факту это два РАЗНЫХ файла под одинаковым `rel_path` — например, два архива были распакованы в разные подкаталоги, но плагин ошибочно сослался на пересекающиеся пути) → finals совпадают, resolved sources разные → FAIL. Прежняя версия guard, использовавшая префикс `[archive_src.name/]`, дала бы здесь ложный PASS (`raw_dest/archive.zip/extracted/foo/bar.pdf` vs `raw_dest/other.zip/extracted/foo/bar.pdf`) — после apply скрипт реально записал бы оба в один и тот же `raw_dest/extracted/foo/bar.pdf`, второй copy2 затёр бы первый;
- contract-тест `tests/unit/test_validate_machine_plan.py::test_raw_dest_collision`: (a) negative — два `raw_only` с `raw_dest_name="archive.zip"` и разными `archive_src` → exit non-zero; (b) **positive (R6 ложный позитив, который guard на basename давал бы FAIL)** — `raw_only.raw_dest_name="contract.pdf"` + `item.source_path=<inbox>/sub/contract.pdf` (`archive_src=null`) → finals различаются (`raw_dest/contract.pdf` vs `raw_dest/sub/contract.pdf`) → exit 0; (c) negative — `raw_only.raw_dest_name="contract.pdf"` + `item.source_path=<inbox>/contract.pdf` (`archive_src=null`, разные physical файлы) → finals совпадают, sources разные → exit non-zero; (d) **negative (real collision that the prior `[archive_src.name/]`-prefix guard would have missed) — `raw_only` с `raw_dest_name="report.pdf"` от source A (`archive_src=<inbox>/archive_a.zip`) → final dest `raw_dest/report.pdf`; одновременно `item.source_path=<work_dir>/report.pdf` с `archive_src=<inbox>/archive_b.zip`** (source_root=`work_dir`, `rel_path=report.pdf`) → final dest тоже `raw_dest/report.pdf`; resolved sources различаются (разные physical файлы: `<inbox>/archive_a.zip` vs `<work_dir>/report.pdf`) → exit non-zero (старый guard с префиксом `[archive_src.name/]` развёл бы их в `raw_dest/archive_a.zip/report.pdf` vs `raw_dest/archive_b.zip/report.pdf` и пропустил коллизию); (e) positive — все raw destinations уникальны → exit 0; (f) positive (edge) — два `items` с одинаковым `source_path` (один и тот же физический файл, дублирующая ссылка) → exit 0 (resolve совпадает).

## Risks / unknowns / assumptions

1. **Совместимость машинного плана с текущим плагин-флоу**. Допущение: добавление `<plan_basename>.yaml` рядом с markdown-планом не сломает review-фазу (Сюзерен по-прежнему смотрит markdown). Риск: Codex может оказаться нестабилен в генерации валидной YAML по схеме при больших intake (>40 файлов). Митигация: валидация в plan-фазе (`apply_intake_plan.py --dry-run` сразу после генерации); если fail — Codex обязан перегенерировать.
2. **Зависимости Python**. Допущение: `pyyaml` уже есть; `pymupdf`/`Pillow` ставятся через `setup.sh` без переписывания. Риск: на машине пользователя `setup.sh` мог не запускаться после переноса. Митигация: SKILL.md уже имеет фразу про `setup.sh` в предусловиях; явно перечислить новые скрипты.
3. **Поведение `apply_intake_plan.py` при revise**. Reference-скрипт **не** делает idempotent skip — повторный apply одного и того же plan-yaml на чистом state ловит `doc_id already exists in index` и возвращает ApplyError (ненулевой exit). Idempotent rerun возможен только из сохранённого `state_file` (если предыдущий apply прервался между staging и promotion). После revise plan-фаза должна перегенерировать `<plan_basename>.yaml` под актуальный `next_id_start` — иначе apply упадёт на mismatch. Это поведение фиксируется test_apply_then_rerun_blocks.
4. **Нет reocr-skill в CC**. `render_pages.py` и реокр-флоу из reference сознательно НЕ переносятся (YAGNI: пользователь о них не спрашивал, в CC такой скилл отсутствует). Если позже потребуется — отдельный plan.
5. **Нет аналитических `validate_opus_output.py`**. CC использует Codex вместо Opus subagent для аналитики (analytical-reviewer.md), и валидация формата уже встроена в Codex prompt. Перенос `validate_opus_output.py` не оправдан без переархитектуры аналитических скиллов — out of scope.
6. **`apply_intake_plan.py` в reference обслуживает все три скилла (intake/add-evidence/add-opponent)**. Маршрутизация идёт через явное `items[].source` ("client"/"opponent"/...) и `origin{}` в каждом item, а НЕ через верхнеуровневое поле `skill` (его в контракте `_validate_plan` нет). Допущение copy-as-is принимается: plan-фаза каждого скилла генерирует одни и те же ключи, отличаясь только дефолтным значением `items[].source` и составом `origin`. Митигация лишних ожиданий: `shared/plan-schema.yaml` НЕ объявляет `skill`. Риск переноса: некоторые plagin-CC поля frontmatter/index (`parties`, `summary_draft`, печать/подпись/комплектность) скрипт не пишет — это решено сужением контракта (см. Contracts → Машинный план), что является сознательным изменением скиллов плагина. Если бизнес-требование сохранить эти поля окажется блокирующим — отдельный план: расширить `_validate_plan` и `_build_index_payload` в `apply_intake_plan.py` с unit-тестами, тогда unknown-key поля становятся валидными extras.
7. **`update-index` apply остаётся LLM-driven**. Сознательное решение: режим A добавляет файлы по одному, отдельные операции мелкие, плюс apply содержит атомарные требования (mktemp/trap), которые скрипт уже инкапсулирует в `prepare_intake_workdir.py`/`apply_intake_plan.py` для intake. Полная скриптовая замена update-index apply потребовала бы ещё одного скрипта вида `apply_update_index.py` — заплашируем только если после Ф2 обнаружится высокая стоимость. Сейчас YAGNI.
